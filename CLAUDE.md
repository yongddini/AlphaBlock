# AlphaBlock — Claude Code 개발 가이드

오더블록(Order Block) 기반 암호화폐 자동매매 프로젝트. **이 저장소에서 Claude Code의 역할은 "개발자"다.** 완료 판단·상태 관리·다음 작업 제안 등 PM 업무는 별도 자동화(Cowork PM 러너)가 담당하므로 여기서 신경 쓰지 않는다.

## 역할 분담 (중요)

- **Claude Code (여기, 로컬)** = 개발. 아래 "개발 워크플로우"만 수행한다.
- **PM 러너 (Cowork, 매시간)** = 완료 판단(In Review→Done), 상태 관리, 다음 Todo 제안, 보고. 코드는 건드리지 않는다.
- 승인은 사용자가 이슈를 **Approved** 상태로 옮기는 것으로 표현된다.

## 토대 확정 (Foundation Freeze) — 2026-07-15

📌 **아래 3개 토대는 확정이며, 그 위에서만 전진한다.** 모든 측정·최적화·전략 이슈는 이
토대를 **고정 입력**으로 받는다. 토대를 바꾸면 그 위에 쌓인 모든 숫자·결정이 무효화되므로,
**변경은 반드시 명시적 「재-베이스라인」 이슈로만** 한다(지나가는 김에 바꾸지 않는다).

1. **진입 방식 = 존 근단 지정가 + 오프셋 2bp**(`entry_mode="zone_limit"`, WAN-95 /
   `zone_limit_offset_bps=2.0`, **WAN-112**). 사용자의 실매매. 진입가를 재산정하는 **볼린저
   밴드의 표본 시점은 `intrabar_live`(봉내 라이브)** 다(**WAN-132** — 아래 📌 문단).
2. **공식 체결 렌즈 = `baseline` 단독**("닿으면 체결", WAN-104 / **🔁 3렌즈 병기 폐지 WAN-128**).
   모든 채택 판단·리포트의 주 수치는 `baseline`으로 내고, **신규 리포트는 `baseline` 단독**이다
   — `pen_5bp`·`pen_5bp_drop_50` 병기 요구는 WAN-128이 폐지했다(사용자 결정, 아래 📌 WAN-128
   문단). 두 민감도 렌즈는 **삭제가 아니라 옵트인**이다(`--fill`로 언제든 수동 확인).
3. **비용 = 현재 값 확정**(테이커 4bp / 메이커 2bp / 슬리피지 5bp, WAN-95 배선 기준).
   비용 현실화(WAN-92)는 **명시적 재-베이스라인**으로만 반영하고, 그 전까지 현재 값이 정본이다.

⚠️ **작업 층 순서(아래층이 굳은 뒤 위층)**: (1) 엔진 정확성/버그 → (2) 토대(위 3개) →
(3) 구조 결정(채택 TF·숏 온오프·오프셋·다중 포지션) → (4) 측정·최적화(익절 R·MFE 등).
**하위층 이슈는 자기 상위 입력이 열려 있는 동안 Approved로 올리지 않는다**(PM 러너는 이
관계를 `blockedBy`로 강제한다). 마지막 구조 잠금이던 **채택 TF는 WAN-107이 닫았다**
(15m·1h 공동 작업 TF — 아래 📌 TF 문단). 그 재판정이 15m을 되살리자 **WAN-103의 다중
포지션 판정도 WAN-108이 15m 축에서 다시 냈고**(결론: 기본값 유지 + 사이징 2안 기각 —
아래 ⚠️ 다중 포지션 문단), 이로써 구조 결정 층은 **닫혔다**. 측정·최적화 이슈(WAN-90 등)는
**15m·1h 두 축을 병기**해야 한다.

이 워크스페이스는 Linear로 이슈를 관리한다. Linear MCP를 연결해 사용:

```

```

## GitHub 연동

원격: `github.com/yongddini/AlphaBlock` (기본 브랜치 `main`). Linear ↔ GitHub 기본 연동을 사용해 **GitHub 동작이 Linear 상태를 자동 동기화**한다: 브랜치/커밋 → In Progress, PR 열림 → In Review, PR 머지 → Done. 따라서 개발자는 상태를 손으로 바꾸기보다 브랜치·PR·머지 행위로 상태가 흐르게 한다. 이슈 연결은 브랜치명 또는 PR 제목/본문의 이슈 식별자(`WAN-x`)로 이뤄진다.

`main`은 브랜치 보호로 **CI(WAN-10) green + PR** 없이는 머지되지 않는다(설정 스크립트: `scripts/setup-branch-protection.sh`, 적용은 저장소 소유자 1회 실행). 머지 버튼은 사람이 누른다 — 자세한 규칙은 README "브랜치 보호 · 머지 정책" 참고.

## 개발 워크플로우

1. **Approved 이슈만 개발한다.** Linear에서 team=Alphablock, state=`Approved` 이슈를 우선순위 순으로 가져온다. Backlog/Todo/Rejected 는 절대 개발하지 않는다.
2. **착수 = 브랜치 생성.** 이슈의 Linear 제안 브랜치명(이슈 식별자 포함, 예 `yu04038/wan-7-...`)으로 `main`에서 브랜치를 만든다. 브랜치·첫 커밋이 올라가면 Linear GitHub 연동이 이슈를 자동으로 **In Progress**로 옮긴다(안 되면 수동 전환).
3. 이슈의 완료 기준(Acceptance Criteria)을 충족하도록 이 저장소에서 개발한다.
4. 커밋 메시지 앞에 이슈 식별자를 붙인다. 예: `WAN-7: 오더블록 탐지 로직 추가`.
5. **완료 = 브랜치 push + PR 생성.** 브랜치를 `origin`에 push하고 `main`으로 향하는 GitHub **Pull Request**를 연다.
   - PR 제목/본문에 이슈 식별자(`WAN-7`)를 넣어 이슈에 자동 연결한다. PR 본문에는 변경 파일·완료기준 체크리스트·테스트/품질게이트 결과를 적는다.
   - PR이 열리면 Linear 연동이 이슈를 자동으로 **In Review**로 옮긴다(연동이 상태를 관리하므로 직접 Done으로 옮기지 않는다).
   - CI(WAN-10)가 PR에서 자동 실행된다. **CI 초록불이 아니면 리뷰 대상이 아니다** — 실패를 고쳐 다시 push한다.
6. PM 러너가 미흡하다고 판단해 **Approved로 되돌리고** 변경요청 코멘트를 달면, 그 피드백을 반영해 **같은 브랜치/PR에 추가 커밋**하고 다시 3~5를 수행한다.
7. **머지는 사람이 한다.** PM 검증 + CI green 확인 후, 사용자가 PR을 `main`에 머지한다. 머지되면 Linear 연동이 이슈를 자동으로 **Done**으로 옮긴다. **Claude Code는 PR을 자동 머지하지 않는다.**

## 안전 규칙

- **실거래 자동 활성화 금지.** `ALPHABLOCK_LIVE_TRADING` 기본값은 `false`. 실주문/자금 이동 코드는 사용자가 명시적으로 승인·수행하기 전까지 활성화하지 않는다.
- API 키·시크릿은 코드에 하드코딩하지 않는다. `.env`(커밋 금지, `.gitignore` 포함)나 환경변수로만 주입하고, 새 설정은 `.env.example`에 예시를 추가한다.
- 테스트가 실패하거나 완료기준을 못 채운 상태로 PR을 열지 않는다(= In Review로 넘기지 않는다).
- **PR을 자동으로 머지하지 않는다.** main 반영(머지)은 CI green + PM 검증 후 사용자가 수행한다.
- `.github/`(CI·릴리즈 워크플로우)는 승인된 이슈 범위 내에서만 수정한다.

## 전략 규칙 요약 (WAN-81 메인 엔진)

컨플루언스 전략(`strategy/confluence.py`)의 확정 규칙. **WAN-23/WAN-66 규칙(EMA 60 +
VWMA 100 익절, 롱 온리, 존당 첫 탭 1회)은 WAN-81로 전면 교체됐다** — 아래가 현재
기본값(`ConfluenceParams()`)이다.

📌 **공식 체결 기준선 = `baseline`(WAN-104 확정, WAN-97 결정 1 번복)**:
[`docs/decisions/wan104.md`](docs/decisions/wan104.md) §1. **모든 채택 판단과 리포트의
주 수치는 `baseline`**("닿으면 체결")로 낸다. 근거(사용자 지시): 지정가 **오프셋**이
"가격을 미리 밀어 관통을 주문 가격으로 만족시키는" 수단인데 그 위에 `pen_5bp`(관통 벌점)를
또 얹으면 **같은 마찰을 두 번 친다**. 사용자의 실매매 = 지정가 걸고 닿으면 체결 =
`baseline`이다. 🔁 **세 값 병기 요구는 WAN-128이 폐지했다 — 재인용 금지**(아래 📌 WAN-128
문단이 정본). 원문은 "세 값을 **`baseline`(공식)** · `pen_5bp`(민감도) · `pen_5bp_drop_50`
(스트레스) 로 병기한다"였으나, **신규 리포트는 `baseline` 단독**이다. 두 민감도 렌즈는
**폐기가 아니라 옵트인**이다(파라미터·`--fill` 존치, 수동 확인 가능).
⚠️ **`baseline`이 낙관이라는 성질은 사라지지 않았다** — 큐 우선순위를 모델링하지 않으므로
**공식 수치는 상한으로 읽는다**(민감도 병기가 없어졌으니 그 상한이 얼마나 부풀려졌는지
자동으로 보이지 않는다 — 아래 📌 WAN-128 「잃는 것」 1·2). 실제 해소는 틱·호가 데이터
(WAN-98) 소관이다.

📌 **체결 민감도 렌즈 병기 폐지(WAN-128 = 사용자 재-베이스라인): 신규 리포트는 `baseline`
단독이다.** [`docs/decisions/wan128.md`](docs/decisions/wan128.md). 사용자 결정("둘 다 버리고
baseline만"). **리포트 층위의 변경이고 엔진은 안 건드린다** — `fill_penetration_bps`(기본
`0.0`)·`fill_dropout_rate`(기본 `0.0`)·`fill_dropout_seed`와 `backtest.run --fill` 옵션은 전부
**존치**한다(파라미터를 지우면 과거 리포트 재현이 깨지고 되돌리기 어렵다). CLI 기본값은 이미
`baseline` 단독이라(`harness.BASELINE_FILL`) 바꿀 코드 경로가 없었고, 3렌즈는 각 기존 리포트
모듈의 `LENS_NAMES` 상수일 뿐이라 **기존 md/csv는 무수정 보존**한다(당시 기록). 신규 리포트만
`baseline` 단독으로 낸다.
- ⚠️ **이 결정으로 잃는 것(기록 — 나중에 "왜 안 보이지"를 막기 위해, PM이 반대 근거로 제시했고
  사용자가 알고 선택했다)**: (1) `baseline`의 **낙관 성질은 그대로**인데 그 상한이 얼마나
  부풀려졌는지 자동으로 볼 장치가 없어진다. (2) **채택 엔진 체결 취약성 실측**(WAN-124 3단)이
  리포트에서 사라진다 — 15m OOS에서 관통을 요구하면 **거래 94.29%가 남는데 수익은 3.30%만
  남는다**(1h 95.46%/35.10%). 즉 수익 대부분이 "스치듯 닿고 되돌아선 체결"에 실려 있다는 사실이
  안 보인다. (3) **결론을 가른 선례**가 끊긴다 — WAN-88의 `baseline` 유의 4셀은 `pen_5bp` 한
  번에 전멸했고 WAN-124는 11→10셀로 살아남아 "성질이 다르다"는 판정이 가능했다(WAN-120은
  `pen_5bp` 부호 반전으로 WAN-119의 잘못된 주장 하나를 걷어냈다). (4) **체결률 81%가 전부
  `baseline` 위의 값**이라(WAN-123 게이트 제거 후) 낙관 가정의 비중은 준 게 아니라 늘었다.
  (5) 기존 3렌즈 md/csv와 신규 `baseline` 단독 리포트는 **열이 안 맞는다**(대조 축 단절).
- ⚠️ **「엣지 없음」(WAN-84/88/111/114/124)이 뒤집혔다는 뜻이 아니다** — 렌즈 병기를 그만둔
  것이지 새 수치를 낸 게 아니다. 큐 우선순위 걱정(`baseline`은 상한)은 그대로이고, 실제 해소는
  틱·호가 데이터(WAN-98) 소관이다.
- **「선택 사항」(리포트마다 `pen_5bp` 잔존율 한 줄)은 넣지 않았다** — 이슈가 **사용자 결정
  없이 개발자가 임의로 넣지 말라**고 명시했고, 자율 루틴에는 그 결정 입력이 없어 보수적으로
  **미구현**했다(추가는 언제든 별도 이슈로 가능). wan128.md §선택-사항 참고.
- **기본값·토대는 바꾸지 않았다**(`ConfluenceParams()` 불변, 실거래 보류 `ALPHABLOCK_LIVE_TRADING
  =false` 유지). 파급: WAN-126(다중TF 겹침)·WAN-127(익절 Phase 2)의 "3렌즈 병기" 사양이
  `baseline` 단독으로 축소된다(격자가 줄어 오히려 유리).
📌 **채택 TF = 15m·1h 공동 「작업 TF」 · 4h 보류 · 1d 제외 (WAN-107 확정, WAN-97 결정 2
부분 개정)**: [`docs/decisions/wan107.md`](docs/decisions/wan107.md). 「작업 TF」 = **후속
최적화·검증을 이 TF 위에서 돌린다**는 뜻이지 실거래 승인이 아니다. **후속 이슈는 15m·1h를
병기한다**(1h는 MDD가 가장 낮고 1분봉 서브스텝 비용이 15m의 ~1/10이라 반복 실험의 기본 축,
15m은 OOS 수익·표본이 가장 큰 축). 4h는 **OOS 심볼당 15.7거래**로 WAN-84 유효 기준 20건
미달이라 **판정 불가**(대조군으로 병기), 1d는 표본(10거래)·수익(+1.02%) 모두 의사결정
가치가 없다.

⚠️ **WAN-97의 「15m 제외」는 번복됐다 — 그 근거(관통 민감도 10.3× · OOS 심볼 편중)를
재인용하지 말 것.** 두 근거 모두 **15m을 1h와 가르지 못한다**(wan107.md §2·§3):
(1) **관통 민감도**의 10.3× vs 5.6×는 **전 구간** 값이고 **OOS에서는 9.6× vs 8.2×로 붙는다**
— 게다가 그 벌점을 다 물려도 `pen_5bp` OOS에서 15m(+11.38%)이 1h(+5.71%)를 이긴다.
(2) **OOS 심볼 편중**은 `baseline`에서 **사실이 아니다** — "15m BTC 마이너스(−3.81%)"는
`pen_5bp` 수치이고 `baseline`에선 **+4.14%**, "어느 TF도 OOS 3/3이 아니다"도 **15m이 3/3**
이라 거짓이다. ETH 편중·IS→OOS 순위 뒤집힘은 참이지만 **1h·4h에 똑같이 적용된다**(ETH를
빼면 15m +9.16%(2/2) > 1h +4.01%(1/2)).
⚠️ **단, 관통 민감도의 진단 자체는 살아 있다** — 수익이 "스치듯 닿은 체결"에 몰린다는
성질은 여전하고, 다만 **15m만의 것이 아니다**(1h도 OOS에서 수익의 30%가 관통 요구 한 번에
사라진다). 실제 해소는 틱·호가 데이터(WAN-98) 소관이다.
⚠️ **이 재판정은 렌즈 교체의 산물이 아니다** — 15m은 **OOS에서 세 렌즈 전부** 1h를 이긴다
(`baseline` +15.09%p, `pen_5bp` +5.68%p, `pen_5bp_drop_50` +3.88%p). 즉 **렌즈를 안 바꿨어도
15m이 앞섰다**. WAN-104의 오프셋 판정이 렌즈에 통째로 의존한 것과 **정반대 성격**이니
둘을 같은 부류로 묶지 말 것.
⚠️ **어느 TF도 "엣지가 있다"는 뜻이 아니다** — 공식 렌즈(`baseline`) OOS는 15m +23.26%(3/3),
1h +8.18%(2/3), 4h +2.85%(2/3)이고, 스트레스(`pen_5bp_drop_50`)에선 15m +3.94%(1/3) ·
1h +0.05%(1/3)로 사실상 0이다. **가장 덜 나쁜 후보**일 뿐이며 실거래 승인이 아니다
(`ALPHABLOCK_LIVE_TRADING=false` 유지, [WAN-86 결정 2](docs/decisions/wan86.md)).
⚠️ **`--years 3`은 마지막 봉 기준으로 창을 자르므로 새 캔들이 쌓이면 창이 미끄러진다**
(`backtest/harness.py:load_recent`) — 옛 리포트 CSV는 비트 단위로 재현되지 않는다(WAN-107
실행에서 WAN-97 대비 +8~12시간 이동, IS만 최대 0.68%p 변동·OOS 무변). 창을 못 박으려면
`--start`/`--end`를 쓸 것.

📌 **엣지 판정(WAN-88 확정): 채택 기본값에 통계적 엣지는 확인되지 않았다.**
🔁 ⚠️ **이 판정의 범위를 정확히 읽을 것 — 무력화 축이 「RSI 게이트」다.** WAN-124가 같은 자로
**볼린저**를 무력화 축에 놓고 다시 재자 **15m에서는 무작위와 구분됐다**(9/12 셀, 아래 📌
WAN-124 문단). **모순이 아니라 다른 질문**이고, 그쪽에도 「엣지 찾았다」로 읽지 말라는 경고가
세 겹 붙는다. 📌 **그리고 이 모듈의 널은 게이트가 없는 새 기본값에서는 돌릴 수 없다** — 이
표의 **대조군 풀이 곧 WAN-123 채택 기본값**이기 때문이다(같은 문단).
`backtest/reports/wan88_long_only_summary.md`(재현: `uv run python -m
backtest.wan88_long_only_validation`). `pen_5bp`에서 **유효 셀 16개 중
p≤0.05 & 실제>무작위평균인 셀은 0개**(OOS만 보면 7셀 중 0개)다. 즉 WAN-84(숏 포함)의
「엣지 없음」 결론은 **롱 온리 + 지정가 + 첫 탭 면제 교정 엔진에서도 유지된다** —
WAN-87의 숏 제거는 절대 수익률을 크게 끌어올렸지만(음수 셀 대거 양전환) **판정은
바꾸지 못했다**. 숏이 롱의 엣지를 가리고 있던 게 아니라 드러날 엣지가 없었다는 뜻이다.
⚠️ **대조표의 "유의 셀 0→4개"를 단독 인용하지 말 것** — 그 4개는 `baseline`(낙관) 축에서만
유의하고 **전부 15m이며, 관통 요구(`pen_5bp`) 한 번에 4개 전부 유의성을 잃는다**(거래는
평균 6.7%만 감소). 유의성이 "지정가를 스치듯 닿고 되돌아선 체결"에 통째로 기대고 있다는
뜻이고, 이는 WAN-97 §2-2가 수익률로 관찰한 관통 민감도를 **유의성 지표로 재확인**한 것이다.
⚠️ **이것을 「15m 제외」의 보강 근거로 읽지 말 것 — 그 제외는 번복됐다**(WAN-107, 위 📌 TF
문단). 관통 민감도 **진단**은 살아 있으나 그것이 **15m만의 성질이 아님**이 확인됐으므로,
이 관찰은 "15m을 빼라"가 아니라 **"공식 렌즈(`baseline`) 수치는 어느 TF든 상한으로 읽어라"**
로 읽는다. **이 판정은 "전략이 손실"이라는 뜻은 아니다** —
`pen_5bp` 1h OOS는 3/3 심볼 플러스(BTC +7.14%, ETH +17.43%, SOL +4.08%)지만, 그 수익이
**같은 오더블록에 무작위로 진입해도 나오는 수준**이라 값이 매겨지는 대상은 RSI·타이밍
규칙이 아니라 오더블록 존 자체이거나 시장 베타일 수 있다.

⚠️ **볼린저 진입가 × 지정가의 채택 규칙(WAN-95)**: 둘 다 켠 조합에서 **볼린저가
이긴다** — 지정가를 존 근단이 아니라 **볼린저가 재산정한 가격에 건다**. 코드 근거:
`backtest/zone_limit_backtest.py`의 `build_zone_limit_candidates`가
`limit_price = params.zone_limit_price(ob)`로 존 근단을 잡은 **직후**
`deviation_filter`가 켜져 있으면 `limit_price = deviation_entry_price(...)`로
덮어쓴다(밴드가 존보다 불리하면 그 셋업은 진입 자체를 건너뛴다 — WAN-75 규칙 3).
즉 "사전 지정가 vs 사후 재산정"의 모순은 없다: 밴드 값은 **탭 봉 시점에 확정**되고
그 가격으로 주문이 걸린 것으로 취급한다. 이 조합은 WAN-95 이전에도 기본값은 아니었지만
**미검증 경로는 아니다** — WAN-70/73/76/84 검정이 모두 `zone_limit + realtime` +
기본 `deviation_filter`로 돌았다.
그 대가로 남는 알려진 한계: 밴드는 탭 봉의 SMA20(= 그 봉 종가 포함)으로 계산되는데
체결은 그 봉 **내부**에서 일어날 수 있다(같은 봉의 종가를 알아야 계산되는 가격에
그 봉 도중 체결). 엄밀히 룩어헤드다 — WAN-95는 이 성질을 **바꾸지 않고 기록만 했다**
(바꾸면 WAN-70/84 검정의 엔진 정의가 흔들리므로 별도 이슈에서 판단한다).
🔁 **WAN-95의 「실무 영향은 SMA20 한 틱 차이 수준이라 작다」는 측정으로 반박됐다 —
재인용 금지**(WAN-115/119, 위 📌 두 문단이 정본). 전제는 먼저 WAN-114가 흔들었고
(볼린저가 여러 부품 중 하나가 아니라 **유일한 플러스 부품**이었다), WAN-115가 밴드 봉 기준을
바꾸자 **15m OOS 증분이 −4.88%p 움직이며 채택 기본값의 부호까지 뒤집혔다**(+3.91% → −0.97%).
각주 크기가 아니다.
⚠️ **단 그 −4.88%p를 「룩어헤드의 몫」이라 부르지 말 것 — WAN-119가 개정했다.** 미래 정보만
빼고 현재 봉은 쓰는 `intrabar_live`는 오히려 **+0.50%p 번다**(15m OOS +4.41%). 즉 룩어헤드를
빼는 것 자체는 **비용이 0에 가깝고**, `prev_closed`가 잃은 건 **현재 봉의 실재하는 정보**였다.
🔁 **그리고 WAN-132가 그 「별도 이슈에서 판단한다」를 닫았다 — 기본값이 `intrabar_live`다**
(아래 📌 문단이 정본). 즉 **위 문단이 기록한 룩어헤드는 채택 경로에서 사라졌다**.
**밴드 정의는 네 가지가 존재한다**(`DeviationFilterParams.band_bar`: `"intrabar_live"`
**기본값** = WAN-119/132(미래 없음 · 현재 봉 씀 = 트레이딩뷰/실매매) · `"tap"` = WAN-131까지의
기본값(룩어헤드) · `"prev_closed"` = WAN-115(미래 없음 + 현재 봉도 버림) ·
`"intrabar_causal"` = WAN-120(직전 1분봉 종가 = 잔여 1분마저 0)). 옛 수치를 결론에 박아 둔
리포트는 `harness.LEGACY_BAND_BAR`(= `"tap"`)로 **명시 고정**돼 있다.
⚠️ **`intrabar_causal`는 정본 후보가 아니라 감사 도구다** — B안(지정가) 전용이고 **A안은
ValueError로 거부한다**(봉 단위에 직전 1분봉 종가가 없다). 정본으로 올리면 대시보드·
`live.runner`·스윕이 깨진다(WAN-120 §4).

**지정가 확정 순서(WAN-99)**: `zone_limit_price(ob)`(존 근단) → `deviation_entry_price`

순서가 강제되지 않는다. 여기서 확정된 가격이 1R(진입가→무효화 경계)과 고정 R 익절
목표의 기준이다.

것이고 그 반대가 아니다 — 사용자 확정 사항이다.

헤더에 경고를 명시했다. WAN-96의 「15m 제외」는 **최악 가정(관통 5bp + 50% 탈락)에서
15m이 −5.8%(0/3)로 무너진다**는 근거였는데, 교정 엔진에서 그 축만 다시 돌리면
**15m +3.57%(3/3), 1h +8.34%(3/3)로 부호가 뒤집힌다** — 즉 제외·보류의 근거가 성립하지
않는다(`wan100_first_tap_free_summary.md` §4). **그렇다고 15m 채택은 아니다**: 15m은
시드 분산이 커(BTC −7.7%~+17.4%) 평균 +3.57%에 개별 마이너스 시드가 섞여 있고,
"수익이 스치듯 닿은 체결에 몰린다"는 WAN-96의 **진단 자체는 교정 후에도 살아 있다**
(`pen_5bp`만 걸어도 15m 수익의 63%가 사라진다: +82.3%→+30.6%, 거래는 6%만 감소).
WAN-97이 IS/OOS 축까지 포함해 재판정했고([`docs/decisions/wan97.md`](docs/decisions/wan97.md),
원본 `wan97_tf_fill_grid.csv`·`wan97_tf_fill_oos.csv`), 결론은 **당시 공식 기준선
`pen_5bp` · 15m/1d 제외 · 1h 작업 TF · 4h 보류**였다. ⚠️ **그 두 축은 이후 모두 번복·개정됐다**
— 렌즈는 WAN-104가 `baseline`으로 번복했고, **「15m 제외」는 WAN-107이 번복해 15m을 1h와
함께 작업 TF로 되돌렸다**(위 📌 TF 문단이 정본). 즉 **이 문단의 「15m 채택은 아니다」는
더 이상 현재 규칙이 아니다** — 당시 서술로만 읽을 것. WAN-99 CSV는 회귀 테스트가 고정한
1셀만 교정 엔진으로 갱신돼 **엔진이 섞여 있으니 셀 간 비교 금지**.

⚠️ **체결 가정 보수화 결과 — 15m 제외 권고(WAN-96, 🔁 폐기됨)**: `backtest/reports/
wan96_fill_conservatism_summary.md`(재현: `python -m backtest.wan96_fill_conservatism_report`).
**아래 수치는 전부 WAN-100 교정 전 엔진 기준이고, 그 권고(15m 제외)는 이제 결론까지
번복됐다** — 근거는 WAN-100 교정에서 소멸했고(아래), 결론은 WAN-107이 뒤집었다(위 📌 TF
문단이 정본). **이 문단은 당시 기록이며 인용 대상이 아니다.**
WAN-95의 15m 반전은 **"닿으면 체결"이라는 낙관 가정에 의존**한다. 체결 규칙을 보수화하면
15m 평균 수익률이 +15.8%(3/3 플러스) → **−5.8%(0/3 플러스)**(관통 5bp + 50% 탈락)로
무너진다. 1h도 같은 최악 가정에서 +9.1% → −1.7%(0/3)다.

핵심 근거는 거래 수와 수익의 비대칭이다: **`pen_5bp`는 거래를 4.7%만 줄이는데
(461→439) 수익은 15.8%→1.3%로 사라진다**. 즉 수익이 **지정가를 스치듯 닿고 되돌아간
체결**(관통 없는 체결)에 몰려 있는데, 그 체결이야말로 실거래에서 큐 우선순위 때문에
**가장 안 될 가능성이 높은** 체결이다. 반면 체결 편향 진단은 체결군이 미체결군보다
**나쁘다**고 나온다(15m −0.18R) — 모순이 아니다. 그 진단은 공통 가상 진입가(탭 봉
종가)로 셋업의 방향성만 재므로 지정가의 가격 이점을 뺀 값이다. 둘을 합치면: 지정가
진입의 수익은 **셋업이 좋아서가 아니라 더 좋은 가격에 들어가서** 나오고, 그 가격은
체결 가정을 조금만 현실화하면 사라진다.

보수화 옵션은 **파라미터로만 존재하고 기본값은 바꾸지 않았다**
(`fill_penetration_bps=0.0`, `fill_dropout_rate=0.0`, `fill_dropout_seed=0`) — WAN-95
결과가 그대로 재현돼야 비교가 성립하기 때문이다. 탈락률 0이면 난수를 뽑지 않으므로
기본 실행은 WAN-95와 비트 단위로 동일하다. 이슈의 (b) 큐 모델(거래량 대비 체결 확률·
부분 체결)은 1분봉 거래량으로 큐 위치를 복원할 수 없어 **구현하지 않았다** — 틱·호가
데이터가 필요하다.

📌 **오프셋 2bp 채택(`zone_limit_offset_bps=2.0`) — WAN-112가 WAN-104 결정 2를 번복**:
[`docs/decisions/wan112.md`](docs/decisions/wan112.md). 사용자 결정("baseline에 offset 2bp").
공식 렌즈는 `baseline` 그대로 유지하고, **체결 신뢰성 보험으로 2bp를 고정**한다.

⚠️ **「데이터가 2bp를 골랐다」로 인용 금지 — 정반대다.** 순수 수익 최대화로는 **0bp가 이기고**

0.43%p를 **체결 신뢰성과 맞바꾸는 사용자 판단**이다 — 실거래에선 큐 우선순위 때문에 "스치듯
닿은 체결"이 가장 안 될 가능성이 높은데 하필 수익이 거기 몰려 있고(WAN-96 비대칭), 2bp가 그
체결을 관통 체결로 바꾼다. 실제로 체결률이 오르지만 **폭은 작다**(IS 49.16%→49.78%, OOS
47.53%→48.48%). ⚠️ **OOS에서 2bp가 이기는 것(+8.53% vs +8.18%)을 근거로 쓰지 말 것** —
노이즈와 구분되지 않고, **OOS는 선택 축이 아니다**. ⚠️ 이 대조는 **1h 전용**이다(WAN-104 격자가
1h만 돌았다) — 15m 축의 교환비는 아직 안 쟀다.

⚠️ **단위 함정: 2bp는 `2.0`이지 `0.0002`가 아니다**(필드가 bp 단위, `/10_000`으로 나눈다).
WAN-112 이슈 본문의 "0.0 → 0.0002"는 분수 표기이고, 그대로 넣으면 **사실상 무효과**라 결정을
반영했다고 믿으면서 0bp를 돌게 된다(WAN-91/95와 같은 부류의 조용한 실패). 회귀 테스트가 막는다.

⚠️ **`build_params`/CLI 격자의 `offset_bps=None`은 "채택 기본값에 맡긴다", `0.0`은 "0bp를
달라"다** — 가르지 않으면 CLI 기본 실행만 혼자 옛 엔진(0bp)을 돈다. 단 **종가 진입이 섞인
격자는 기본이 0bp**다(A안은 오프셋을 안 읽으므로 두 팔을 맞춰야 진입 방식이 격리된다).

(`backtest/portfolio.py`)와 포트폴리오 레버리지(= **열린 명목 합** ≤ 자본 × leverage)를
만들었지만, **`ConfluenceParams()`도 `risk_sizing`(사이징 모드·레버리지)도 바꾸지 않았다** —
`PortfolioParams`를 명시적으로 넘기는 새 진입점(`run_zone_limit_portfolio_backtest`)에서만
돈다. 기존 경로는 비트 단위로 그대로다.

📌 **「다중 포지션은 나쁘다」는 일반 결론으로 인용 금지 — WAN-108이 「1h에서 나쁘다」로
좁혔다**(wan103.md 「WAN-108 재판정」 절). **작업 TF 둘이 정반대를 가리킨다**:
- **15m OOS는 다중이 이긴다** — series 심볼평균 **+23.26% → +38.81%(3배)**, 3/3 심볼 플러스,
  수익/MDD도 **2.34 → 2.59**로 개선(ETH 제외해도 +9.16% → +13.03%). pooled 15m은
  +65.93% → +139.82%, 통합 책(15m/1h/4h/1d)은 **+36.09% → +196.79%**.
  ⚠️ **이 수치는 전부 `baseline`(낙관) 전용이다 — 렌즈 표기 없이 인용하지 말 것**(WAN-110):
  `pen_5bp`에서 우위가 +15.54%p → **+2.28%p**로 줄고 「3/3 심볼 플러스」도 **2/3**이 된다.
- **1h OOS는 단일이 이긴다** — **+8.18% → +5.66%(3배)**, MDD **5.96% → 8.94%**, 수익/MDD
  **1.37 → 0.63**. (WAN-103 수치가 그대로 재현됐다 — 갈린 원인은 엔진이 아니라 **TF 축**이다.)
- **기본값 유지 근거**: 하나의 기본값으로 둘 다 좋게 할 수 없고, "TF마다 다른 포지션 정책"은
  **IS에서 고르는 새 자유 파라미터**다. 또 15m의 승리는 `baseline`(낙관) 위에 서 있는데
  다중 포지션은 **체결에 더 의존**하므로(OOS 거래 772→838건) 체결 보수화 축의 재검이
  선행돼야 한다 — 그게 15m 축 후속 이슈의 질문이었다.
  🔁 **WAN-110이 그 재검을 끝냈다: 통과하지 못했다**(wan103.md §F, 재현: `python -m
  backtest.wan110_multi_position_fill_conservatism`). 같은 셋업 풀에 렌즈만 조이면 **15m
  다중 우위가 +15.54%p → +2.28%p(`pen_5bp`)로 마진의 85%가 사라진다**(거래는 5%만 감소).
  WAN-96 비대칭도 **다중에서 더 심하다**(`pen_5bp` 잔존 수익 단일 48.9% vs 3배 35.2%) —
  즉 다중이 더 버는 몫은 유독 "스치듯 닿은 체결"에 몰려 있다. **기본값 유지 결론은
  그대로이고 근거가 하나 늘었다.** ⚠️ 스트레스 렌즈의 부호 역전(−1.62%p)을 "단일이
  이긴다"로 인용하지 말 것 — 시드 5개 중 3개는 오히려 다중이 앞선다(평균을 2개가 끌어내린다).
  읽을 수 있는 건 "다중 우위가 사라진다"까지다.

📌 **사이징 2안(사용자의 「1/N 시드 × N배」)은 기각됐다 — 업사이드가 없다(WAN-108)**:
`execution.sizing`에 `sizing_mode`가 생겼지만 **기본값은 `"risk_pct"`(현행)** 이다.
`"fixed_notional"`(명목 = 자본 × `notional_fraction`, 손절 거리 무관)은 **수익이 더 낮고
MDD가 더 크다**(15m OOS: 1안 3배 +38.81%/MDD 14.98% vs 2안 5배 **+18.89%/MDD 19.61%**;
1h OOS 2안 5배는 **−7.93%/MDD 22.93%**, pooled 1h는 **−25.00%/MDD 48.23%**).
- ⚠️ **2안은 동시 1포지션에서도 진다**(15m OOS single: +23.26% vs **+19.15%**) — 즉 문제는
  겹침·레버리지가 아니라 **사이징 그 자체**이고, 이 기각은 다중 포지션 논쟁과 **독립**이다.
- ⚠️ **「청산 0건」을 2안의 안전 근거로 쓰지 말 것**(wan103.md §5의 경고가 그대로 유효).
  청산은 1안·2안 모두 0이지만 **최대 동시 리스크는 1안 12.0% vs 2안 55.7%(4.6배)** 다.
  1안에서 레버리지가 명목 천장일 뿐이었던 건 **리스크 사이징 덕**이고, **2안에서 레버리지는
  실제 위험 배수**다. 청산이 안 난 건 실측 손절 거리(명목 대비 ~4~8%)가 자본을 다 먹을 만큼
  멀지 않아서다.
- ⚠️ **「역대 최대 겹침 N배」는 최적이 아니다**(15m 포함해 유지): IS에서 잰 N을 OOS에
  적용하면 더 낮은 레버리지에 진다(15m: N=8 +128.87% < 3배 +139.82%; 1h: N=5가 **꼴찌**
  +15.13% < 1배 +20.48%). **OOS 자연 겹침이 IS의 N을 넘긴다**(1h 5→6, multi_tf 11→12) —
  "N=역대 최대"는 미래의 최대가 아니다. N은 시간의 **0.13%**(1h)·**0.004%**(multi_tf)에만
  성립하는 상태다.

⚠️ **비용 가정(WAN-95 기준)**: 테이커 4bp(`BacktestConfig.fee_rate`) / **메이커 2bp**
(`maker_fee_rate`, WAN-95에서 `None`→`0.0002`로 배선 — 예전엔 지정가 진입에도 테이커가
붙었다) / 슬리피지 5bp(테이커 체결에만). **지정가 진입=메이커(슬리피지 0), 청산=항상
테이커**(손절·익절 도달은 시장가 성격). 테이커 4bp→5bp 등 비용 가정 현실화는 WAN-92 범위.

⚠️ **라이브/페이퍼는 아직 지정가를 집행하지 못한다(WAN-45)**: `ConfluenceStrategy`는
`entry_mode`를 읽지 않는 **A안 시그널 생성기**라, 실시간 러너(`live.runner`)는 채택
기본값이 지정가로 바뀐 뒤에도 여전히 종가 기준 시그널·알림을 낸다. 대시보드도 상위TF만
로드하므로 종가 진입(A안)으로 명시 고정돼 있다(화면 배지가 "A안(봉 마감 종가)"로 표시).
즉 **백테스트(채택 규칙) ↔ 라이브(집행 가능 규칙)가 갈라져 있다** — WAN-45(러너 ↔
LimitOrderBook 배선)가 이 간극을 닫는다.

## 실험 도구 — 새 리포트 스크립트를 짜기 전에 읽을 것 (WAN-101)

**"파라미터 바꿔서 다시 돌려보기"에는 새 스크립트를 만들지 않는다.** 범용 CLI가 있다:

```bash
uv run python -m backtest.run --tp-r 1.0,1.5,2.0,3.0          # 익절 R 스윕
uv run python -m backtest.run --tf 15m --fill baseline,pen_5bp,pen_5bp_drop_50
uv run python -m backtest.run --symbol BTCUSDT,ETHUSDT --oos --format csv --out x.csv
uv run python -m backtest.run --tf 1h --oos-warm                          # 따뜻(주)+차가움(스트레스) 병기(WAN-166 정본)
uv run python -m backtest.run --symbol BTCUSDT,ETHUSDT,SOLUSDT --jobs 6   # 병렬(WAN-121)
uv run python -m backtest.run --tf 15m,1h --positions single,3 --oos      # 단일 vs 다중(WAN-130)
uv run python -m backtest.run --tf 15m,1h --combine-obs true,false --oos  # 존 병합 전후(WAN-149)
uv run python -m backtest.run --tf 1h --max-zone-width-atr none,1.28 --oos # 존폭 필터 off/on(WAN-159 기본 1.28)
uv run python -m backtest.run --tf 1h --max-zone-width-atr none,1.28 --tp-r 1.0,1.5,2.0,3.0 --oos  # 문턱×배수(WAN-161)
uv run python -m backtest.run --symbol BTCUSDT --tf 15m --trades t.csv --equity s.csv  # 거래별(WAN-106)
uv run python -m backtest.run --symbol BTCUSDT --tf 15m --persist         # DB 적재 → 대시보드 조회(WAN-106)
```

📌 **거래별 내역·DB 적재(WAN-106)** — 요약 1행은 "언제 샀나 · 어디서 손절났나"에 답하지
못한다. `--trades`/`--equity`는 **단일 조합 전용** CSV(KST·UTC 병기 · 시드 변화 포함)이고
격자면 **거부한다**(조용히 마지막 조합만 내보낸 파일이 나중에 "채택 엔진의 거래"로 인용되는
것이 WAN-95의 교훈). `--persist`는 격자에서도 되며 **거래 · 미체결 셋업 · 시드곡선**을
`backtest/trade_store.py`의 테이블(`backtest_runs`/`_trades`/`_trade_exits`/`_setups`/
`_equity`)에 넣는다. 대시보드 **「저장된 거래」 탭**이 그걸 **계산 없이 조회**한다(손절/익절
필터 · 미체결 셋업 · 행 클릭 → 차트 점프).
- 📌 **실행 지문 없이는 적재도 조회도 되지 않는다** — 지문 = 심볼·TF·구간·창 + `ConfluenceParams`/
  `OrderBlockParams`/`BacktestConfig` 직렬화 + **엔진 버전 + 코드 리비전**(git 해시, 더러우면
  `-dirty`)이고 그 해시가 `run_id`다. **코드 리비전이 핵심**이다: 파라미터만으로 키를 만들면
  엔진 버그를 고쳐도 키가 같아 **옛 결과를 꺼내 준다**(WAN-91/95/112와 같은 부류의 조용한 실패).
  같은 지문 재적재는 **기본이 거부**(`--persist-replace`가 명시적 덮어쓰기).
- ⚠️ **지표의 정본은 적재된 요약이지 복원 결과가 아니다** — 종가(A안) 엔진의 자본곡선은 **봉
  단위**라 거래 단위로 다시 만든 곡선과 **MDD·Sharpe가 다르다**(최종 시드·거래 수·수익률은 같다).
  화면도 요약 지표를 쓴다.
- 컬럼은 **WAN-146이 만든 `backtest.report.trades_to_display_frame`을 그대로 재사용**한다
  (`include_utc=True`가 파일·DB용 옵트인) — 화면·CSV·DB가 두 벌로 갈라지면 같은 거래가 세 곳에서
  다른 숫자로 보인다.

값에 콤마를 주면 데카르트 곱 격자를 돌고 조합별 1행(`total_return`/승률/MDD/거래수/
체결률/평균 R/Sharpe)을 낸다. 축은 심볼·TF·진입 방식·익절 R·오프셋·재탭 정책·**포지션
정책**(`--positions`, WAN-130: `single` = 동시 1포지션 기본값 · 숫자 = 다중 포지션 명목
상한 배수)·**존 병합**(`--combine-obs`, WAN-149 — ⚠️ 이것만 **탐지** 파라미터라 값마다
오더블록을 다시 탐지한다)·**존폭 필터**(`--max-zone-width-atr`, WAN-159 — 채택 기본값 `1.28`,
`none`은 **끄기**이고 인자 미지정(1.28)과 다르다, 단위는 **ATR 배수**다)·체결 가정·시드이며,
`--oos`/`--walkforward`가 IS/OOS 분할을 기본 제공한다. **정본 리포트의 OOS는 `--oos-warm`**
(WAN-166: `oos_warm` 주 수치 + `oos` 스트레스 병기 — B안 단일 포지션 전용, 위 📌 문단)이다. 인자 없이 돌리면 **채택 기본값**
(`ConfluenceParams()`) 그대로이고, 그 결과가 WAN-95/99 리포트 셀과 1e-9 이내로 일치함이
테스트로 고정돼 있다(`tests/test_harness.py` + `tests/test_run_regression_real_data.py`).

📌 **`--jobs N`(기본 1 = 직렬)은 (심볼, TF) 단위 병렬이고 숫자를 바꾸지 않는다(WAN-121)** —
`--jobs`는 **성능 노브이지 결과 축이 아니다**(직렬과 행·순서·stdout·stderr가 비트 단위로
같음을 합성·실데이터 양쪽 테스트가 고정한다). `auto`/`0`이면 `os.cpu_count()`.
⚠️ **기대 배수를 코어 수로 잡지 말 것** — 실측은 **6심볼 1h 3년 격자에서 93.2s → 69.7s
(1.34배)** 수준이다(그 머신의 CPU 바운드 상한 자체가 2.36배였다). 두 가지가 천장을 만든다:
(1) **1분봉 로딩이 1조합 실행의 ~36%** 인데 이건 셀마다 필요해 병렬로 사라지지 않고
(WAN-91 이래 `--start`/`--end` 창은 캐시를 안 타 매번 SQL 전수 스캔이다),
(2) 애플 실리콘은 `cpu_count()`가 **E코어까지 세므로**(M1 = P4+E4) `auto`가 성능 코어 수보다
크게 잡힌다. 격자가 작으면 프로세스 기동 비용(spawn 재임포트)이 이득을 넘겨 **오히려
느리다**(600봉 합성 6심볼에서 2.3s → 4.5s). 조합이 많은 스윕일수록 유리하다.

공통 골격은 `backtest/harness.py`(데이터 로딩·파라미터 조립·경로 스위치·구간 분할·렌더)다.
새 `wanNN_*.py` 리포트는 **CLI로 답이 안 나오는 것**(사후 분해·편향 진단·결론 문장이
필요한 분석)에만 만들고, 그때도 골격은 harness에서 가져다 쓴다. 일회성 스크립트
아카이브는 WAN-94 소관이다.

📌 **신규 리포트는 `baseline` 단독으로 낸다(WAN-128)** — `LENS_NAMES = ("baseline",
"pen_5bp", "pen_5bp_drop_50")` 같은 3렌즈 병기를 새로 짜지 말 것. 공식 렌즈는
`harness.BASELINE_FILL`(= `harness.fill_preset("baseline")`) 하나다. `pen_5bp`·
`pen_5bp_drop_50`은 **삭제가 아니라 옵트인 민감도**라 `FILL_PRESETS`·`--fill`에 그대로 있고,
`uv run python -m backtest.run --fill baseline,pen_5bp,pen_5bp_drop_50`처럼 **수동으로 언제든**
확인할 수 있다(위 예시 블록의 `--fill`). 기존 `wanNN_*.py`의 3렌즈 상수는 **당시 기록이라
손대지 않는다**(무수정 보존 — 그래야 CSV가 비트 단위로 재현된다). 배경·「잃는 것」은 위
📌 WAN-128 문단.

⚠️ `--entry-mode close`에 지정가 전용 노브(`--offset-bps`·`--fill`)를 주면 CLI가
거부한다 — 조용히 무시하면 아무 일도 하지 않은 결과에 라벨만 붙기 때문이다(WAN-95의 교훈).
