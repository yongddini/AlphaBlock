## 요약 — 결론 먼저: (c) 겹침은 유효 선별 규칙이 아니다

공식 렌즈(`baseline`) OOS, 정본 겹침 정의(`contained`), 6심볼 심볼평균:

| | 선별 `B−A` | 가격 `C−B` | 거래수 A→B→C |
| -- | -- | -- | -- |
| **15m OOS** | **−1.13%p** | +4.48%p (오염) | 1681 → 1625 → 782 |
| **1h OOS** | **−0.17%p** | −3.59%p | 507 → 504 → 293 |

1. **선별 효과(`B−A`)가 두 TF·세 구간 모두 0/음수다.** 15m −1.72/+0.35/−1.13(full/is/oos),
   1h −0.48/−0.15/−0.17. 겹침을 요구해도 성과가 안 오른다.
2. **필터가 사실상 아무것도 안 걸러낸다 — 이것이 핵심이다.** A(겹침 무시) 대비 B(겹침 요구)의
   거래 수가 거의 같다(15m 1681 vs 1625 = 96.7%, 1h 507 vs 504 = 99.4%). 상위 존까지 내려가면
   같은 방향 하위TF 오더블록이 **거의 항상 겹치기** 때문이다(바닥 TF 분해: 대부분 15m에서
   이미 겹침). 그래서 `B−A`≈0은 **자명**하다.
3. **매칭 널은 두 TF 모두 퇴화다.** `overlap_fraction`이 95% 이상이라(96.7%·99.4%) `B`가
   풀 `A`와 사실상 같아 부트스트랩이 자기 자신을 검정한다 — WAN-124 퇴화 가드가 그대로 걸린다.
   즉 널을 "돌릴 수 없다"가 아니라 **필터가 너무 약해 검정 대상이 안 된다**는 발견이다.
4. **가격 효과(`C−B`)는 판정 근거가 못 된다.** 15m OOS만 +4.48%p로 튀지만 (a) `B`/`C` 거래
   수가 **52% 다르다**(오염 — 볼린저가 좁은 하위존을 대거 기각) → 순수 가격 효과가 아니고,
   (b) 15m IS는 `C−B` −4.19%p로 **부호가 뒤집힌다**(1h는 IS +2.79 → OOS −3.59). IS→OOS
   불안정이라 과최적화 신호다. 오염 게이트가 자동으로 15m을 (b)에서 **(c)로 내린다**.
5. **심볼 편중 아님.** leave-one-out에서 `B−A`는 어느 심볼을 빼도 ~0을 유지한다(0.05~−0.26%p).
   한 심볼이 만든 착시가 아니다.

⚠️ **`baseline`은 낙관 렌즈(상한)**이고 `C−B`는 볼린저 흡수로 가격 효과의 하한이다 —
이 판정은 그 두 경고 아래에서도 (c)다(선별이 애초에 0이라 렌즈를 조여도 결론이 안 바뀐다).

**= WAN-84/88/111/114/124의 「좋은 존과 나쁜 존을 가르는 선별 규칙이 없다」를 다중TF 겹침
캐스케이드로 재확인.** 채택(재-베이스라인) 권고 없음 — 기본값 불변, 실거래 보류 유지.

> ⚠️ **채택으로 넘어갈 이유가 없지만, 만약 넘어간다면** 그 순간 이 아래 층(WAN-90 익절 등)
> 수치가 전부 재산출 대상이 된다(진입 규칙이 바뀌므로). 지금은 측정 이슈라 파급 없음.

## 1. 무엇을 하는가

사용자 제안(2026-07-17): **"상위TF에서 오더블록 진입이 확인되면, 그 아래 TF에 오더블록이
겹치는지 내려가며 확인하고, 겹치는 자리에서만 진입한다(겹침이 없으면 진입하지 않는다)."**
SMC/ICT 계열의 "상위TF POI + 하위TF refinement"다. WAN-84/88/111/114/124가 반복해서
"좋은 존과 나쁜 존을 가르는 선별 규칙이 하나도 없다"고 냈고, 이 제안은 그 **선별 규칙 후보**다.

**측정·검증만 한다.** `ConfluenceParams()` 기본값도 `build_zone_limit_candidates`의 기본
동작도 바꾸지 않는다 — 겹침은 **옵트인**(`MultiTfOverlapParams` + `zone_provider`를 명시적으로
넘기는 경로)에서만 돈다. 채택은 사용자가 별도 재-베이스라인 이슈로 결정한다.

## 2. 겹침 정의·캐스케이드 사다리·적용 순서 (확정 — 코드와 일치)

사용자 확정 사양(2026-07-19, Linear 코멘트)을 코드에 못박았다. 구현: `backtest/multi_tf_overlap.py`.

* **방향 일치** — 상위TF 수요(BULLISH)에는 하위TF **수요 존**만 겹침으로 센다(반대 방향 제외).
  근거: "같은 자리에서 같은 신호"라는 규칙의 논리. 대가로 표본이 더 깎이므로 미달 셀 수를
  리포트에 명시한다.
* **캐스케이드 사다리** — 상위TF 아래로 내려가며 **첫 겹침에서 멈춘다**. 사양의 바닥은 1분봉:
  1h이면 `1h → 15m → 5m → 1m`, 15m이면 `15m → 5m → 1m`. 사다리는
  `MultiTfOverlapParams.ladder`가 정하고, 데이터 없는 TF는 조용히 건너뛴다.
  ⚠️ **실측 데이터에 5m이 없다** — 아래 §6. 실제 사다리는 `1h→15m→1m` · `15m→1m`이 된다.
* **겹침 정의 3종**(`zones_overlap`) — 정본 `contained`(완전 포함), 민감도 `proximal_in`
  (근단이 상위 존 안)·`touch`(조금이라도 교차). 세 정의는 **중첩 관계**(`contained` ⊂
  `proximal_in` ⊂ `touch`)라 "3개 중 2개 유의"를 독립 증거로 세지 않는다. 헤드라인·결론
  문장은 `contained`로 낸다.
* **first-touched 선택**(`choose_refinement_zone`) — 겹치는 하위TF 존이 여럿이면 가격이
  먼저 닿는 존(롱=근단 최고, 숏=근단 최저)을 고른다. 동률은 거래량↓·확정 시각↓로 결정론적.
* **적용 순서 = 씨앗 존만 갈아끼운다** — 진입 사슬 `zone_limit_price → deviation_entry_price
  (볼린저가 이김) → apply_zone_limit_offset`(WAN-99)은 **세 팔이 글자 그대로 동일**하고, `C`
  팔만 `zone_limit_price`가 읽는 씨앗 존을 상위→하위로 바꾼다. 볼린저를 끄거나 건너뛰지
  않는다(WAN-95 "볼린저가 이긴다" 규칙 유지). 오프셋 2bp(WAN-112)도 그대로 맨 끝에 얹힌다.

### 3팔 (선별 대 가격 분리 — 이 이슈의 본론)

| 팔 | 겹침 요구 | 씨앗 존(진입가·손절·1R 기준) | 무엇을 재는가 |
| -- | -- | -- | -- |
| `A` (대조)     | 없음 | 상위TF 존 | 채택 기본값 |
| `B` (선별만)   | 있음 | 상위TF 존 | 겹침 필터만의 기여 (1R 불변) |
| `C` (선별+가격) | 있음 | **하위TF 겹침 존** | 필터 + 좁아진 1R의 합 |

`B − A` = 순수 선별 효과, `C − B` = 순수 가격 효과. `C`의 손절 = **하위TF 존 무효화 경계**
(그래야 1R이 실제로 줄어 "가격 효과"가 측정된다 — 사양 §4). ⚠️ **알려진 보수적 편향**: 볼린저가
최종 가격을 덮어쓰므로 하위 존 근단이 만든 가격 이점 일부가 볼린저에 흡수돼 `C−B`에 안
잡힌다 → `C−B`는 가격 효과의 **하한**으로 읽는다. ⚠️ `B`/`C`는 볼린저 기각 규칙(WAN-75 규칙 3)
때문에 같은 거래 집합이 아닐 수 있다 → 리포트가 거래 수·체결률을 셀마다 병기하고, 차이가 5%
초과면 `C−B`를 순수 가격 효과로 인용하지 않는다(사양 §3).

## 3. 🚨 룩어헤드 가드 (완료기준 2 — 확정)

상위TF 탭(`signal.trigger_time`) 시점에 하위TF 오더블록을 **그 순간까지의 데이터로만**
판정한다. 겹침은 `ZoneProvider`(시각 → 그 시각 활성 존)에만 의존하고, 기본 구현
`order_block_zone_provider`는 `strategy.models.select_active`로 아카이브를 탭 시각까지
**클리핑**한다(`_clip_to_time`) — 그 시각 이후 확정된 존·아직 없던 무효화/탭은 뷰에서 제외.

회귀 테스트(`tests/test_multi_tf_overlap.py`, 18개)가 이를 **동작으로** 고정한다:
* provider 단위: 탭 시각 이후 확정된 존이 뷰에 나타나지 않음(시각을 미래로 옮기면 그제야 보임).
* 엔진 end-to-end: 겹침 존이 탭 **이후** 확정이면 진입이 안 생기고, **이전** 확정이면 생긴다
  (양성 대조) — 차이를 만든 것은 오직 확정 시점임을 거래 산출로 증명.
* 3팔 배선: `A`=overlap=None과 비트 단위 동일, `B`=겹침 없으면 0진입·있으면 A와 같은 씨앗,
  `C`=하위TF 존으로 손절·1R 재계산.

## 4. 기본값·토대 불변 (확정)

`ConfluenceParams()` 불변, 실거래 보류(`ALPHABLOCK_LIVE_TRADING=false`) 유지. 겹침 파라미터를
안 넘기면 `build_zone_limit_candidates`는 **비트 단위로 예전과 같다**(기존 회귀 테스트
101개 통과로 확인). `_Candidate`에 `refinement_tf`(진단 전용, 기본 None) 한 필드만 추가했고
이는 `Trade`에 실리지 않아 기존 리포트 CSV 재현에 영향이 없다.

## 5. 실행 — 1분봉 탐지 컴퓨트 벽과 캐시 해법 (완료)

1분봉 오더블록 탐지가 **심볼당 8분+**이다(탐지기 `combine`이 매 봉 O(누적존²)이라 봉이
15배(1.58M)면 초선형 폭발 — 3개월 슬라이스 11초에서 선형 외삽한 130초 추정은 틀렸다).
6심볼이면 1분봉 탐지만 ~1시간+이고 이 개발 환경의 백그라운드 작업은 10분 상한이라 한 번에
못 돌린다. ⚠️ **탐지기는 토대라 손대지 않았다**(건드리면 WAN-70/84/88/95/111/114 엔진 정의가
흔들린다).

**해법: LTF 오더블록 아카이브 디스크 캐시**(`detect_ltf_archives(cache_dir=...)`, 기본
`data/cache/wan126_ob` — gitignore). (심볼, TF, 창)별로 한 번만 탐지해 JSON으로 저장하고
재사용한다 — 탐지가 결정적이라 캐시가 정합적이다. 1h 격자가 15m/5m/1m을 캐시하면 15m 격자
(5m/1m 사다리)는 재탐지 없이 백테스트만 돈다. 실행은 detached 프로세스로 냈고(읽기 전용이라
WAL로 `alphablock collect` 데몬과 공존), 1h+15m 252행이 정상 생성됐다.
