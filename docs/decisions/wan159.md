# WAN-159 — 존폭 필터를 채택 기본값으로 (재-베이스라인, 사용자 결정)

**결정(사용자, 2026-07-21)**: `ConfluenceParams.max_zone_width_atr` 기본값을 `None`(꺼짐, 전부
매매) → **`1.28`**(존폭 ÷ ATR ≤ 1.28인 **좁은 존만** 매매)로 옮긴다. WAN-158 결정 **B(옵트인)**
→ **A(채택)** 승격. 사용자 발언: *"우리는 좁은 존을 기본으로 하자는말이야"* → 권고안 제시 후
*"좋아"*.

**측정이 아니라 전환이다.** 존폭 필터가 채택할 값이 있는지는 WAN-142(표본 축소 배제) ·
WAN-152(기하 배제, 선별 성립) · WAN-154(세 자·문턱 3종·체결 보수화 강건성)가 이미 관문 셋을
통과시켰다. 이 이슈는 그 위의 **기본값 이동 + 파급 처리**다. 새 격자는 돌리지 않았다.

## 1. 무엇을 바꿨나

- `strategy/models.py`: `ConfluenceParams.max_zone_width_atr` 기본값 `None` → `1.28`.
  `zone_width_atr_length`는 `14` 그대로.
- `backtest/harness.py`: `LEGACY_MAX_ZONE_WIDTH_ATR = None` 신설(`LEGACY_RSI_GATE_MODE`·
  `LEGACY_BAND_BAR`·`LEGACY_COMBINE_OBS`와 같은 패턴) + `pin_zone_width()` 헬퍼.
- **채택 성과(`wan95_zone_limit_summary.md`)만 재산출**했다(WAN-112/123/132/149와 같은 관례).

### 문턱 1.28을 고른 근거 — 데이터가 아니라 사용자 판단이다

측정 권고 문턱은 **15m 1.24 · 1h 1.32**(출처: `wan154_threshold_stability.csv`의 하위 1/3
분위)로 갈린다. TF별 값을 두면 **앞구간에 맞춰 고르는 자유 파라미터가 하나 는다**(WAN-108이
경계한 자리)라, 두 권고값의 **중간 1.28로 통일**했다. 6종목 문턱이 1.16~1.39에 몰려 있어
중간값의 손해가 작을 것으로 본다. ⚠️ **「데이터가 1.28을 골랐다」로 인용 금지.**

### WAN-155와의 순서 — 이미 어긋났으나 피해 없음

본문은 「이 이슈 먼저 → 그 위에서 WAN-155」였으나 실제로는 WAN-155가 먼저 머지됐다(PR #118).
그러나 WAN-155 모듈이 문턱을 **자기 입력으로 `1.28` 명시**해 돌렸으므로(`wan155_tp_ruler_vs_
multiple.py`) 그 판정은 이미 좁은 존 위에서 나왔고, 이 전환으로 **다시 돌릴 필요가 없다**.

### TRX — 그대로 두고 기록만

TRX 15m은 좁은 존 후보의 **92.6%를 손절폭 가드(0.3%)가 잘라 16거래로 붕괴**한다. 원인은
필터가 아니라 가드다. 가드 변경은 그 자체로 또 하나의 재-베이스라인(WAN-76/79 소관)이라
**손대지 않았다**. 종목 제외도 「성적 나쁜 종목을 뒤늦게 빼기」(WAN-111이 경계)라 하지 않았다.

## 2. 🚨 「이중 필터」 함정과 그 처리

세 모듈(+확장 1)은 좁은 존 팔을 **파라미터가 아니라 후보 리스트를 자기가 걸러서** 만든다:
`wan133_geometry_vs_selection` · `wan142_zone_width_filter_verdict` · `wan152_selection_vs_
geometry` · `wan154_stop_width_audit`(wan152 확장). 기본값이 1.28이 되면 **엔진이 먼저 1.28로
거르고, 그 위에 모듈이 자기 문턱으로 또 거른다** → `default`(전부 매매) 팔이 오염되고,
`matched_random` 개수 맞추기가 깨진다. **이 이슈의 근거가 된 바로 그 표들이 조용히 망가진다.**

**처리**: 네 모듈 전부 `max_zone_width_atr=harness.LEGACY_MAX_ZONE_WIDTH_ATR`(= `None`)로 **명시
고정**했다. 회귀 테스트가 동작으로 잡는다(`tests/test_wan159_zone_width_default.py`).

### `wan155`·`wan161`은 손대지 않았다 — 이미 명시하므로 안전하다

두 모듈은 문턱을 자기 입력으로 명시한다(`max_zone_width_atr=FILTER_THRESHOLD if filter_on else
None`). WAN-159의 센티넬 설계(아래 §3) 덕분에 **`filter_on=False` 팔의 명시적 `None`이 이제
「끄기」로 올바로 해석**되어, 기본값 전환 전후로 **비트 단위 같은 행**을 낸다. wan161의 가드
(`model_copy`로 명시 해제 + 값 불일치 시 예외)도 그대로 유효하다.

## 3. 센티넬 설계 — `None`(끄기)과 「미지정」(채택 기본값)을 가른다

존폭 필터는 **끄기 = 명시적 `None`**이라, `None`을 "손대지 않는다"로 쓰던 `offset_bps` 규약을
그대로 못 쓴다(둘이 충돌한다). 그래서:

- `harness.build_params(max_zone_width_atr=...)`의 인자 기본값을 센티넬 **`UNSET`**으로 두었다.
  `is UNSET`이면 `base`의 값을 물려받고(= 채택 기본값 = `1.28`), 명시적 `None`이면 **끈다**.
  (WAN-159 완료기준 3: `build_params(max_zone_width_atr=None)`은 「끄라」, 미지정은 「채택
  기본값」.)
- CLI 축(`--max-zone-width-atr`): `none` 토큰 → **끄기**(`None`), 인자 미지정 → **채택
  기본값**(1.28). `Grid.max_zone_widths_atr` 기본값은 `(UNSET,)`.
- **`UNSET`은 `enum.Enum` 멤버**다 — CLI 축이 `--jobs` 병렬에서 워커로 **피클**되는데, 평범한
  `object()` 센티넬은 언피클 때 새 인스턴스가 되어 비교가 깨진다(실제로 `--jobs` 테스트가
  이걸 잡았다). Enum 멤버는 이름으로 피클돼 싱글턴이 유지된다.

종가 진입(A안, `entry_mode="close"`)은 이 필드를 읽지 않으므로 `build_params`/`evaluate`/CLI가
**양수 문턱**과의 조합을 거부한다(끄기 `none`/`None`은 무해하므로 허용). A안 param 팩토리
(`CLOSE_ENTRY_DEFAULTS`·대시보드·`ab_run`·wan87/91/95의 close 팔)는 필터를 **명시적으로 끈다**.

## 4. 파급 처리 — 고정한 리포트 목록

**고정 대상을 가르는 규칙**: 리포트가 **자기 엔진을 고정하는가**(`pin_band_bar`/`Grid.band_bar`
=`LEGACY_BAND_BAR` 등으로 밴드를 tap에 못 박은 **동결 스냅샷**)인가, 아니면 **오늘의 채택
기본값을 따라가는가**(밴드를 고정하지 않고 `== ConfluenceParams()`를 테스트로 주장하는 **추적**
모듈)인가. **동결 스냅샷만** `max_zone_width_atr=None`으로 명시 고정한다 — 추적 모듈까지 끄면
그 모듈의 「채택 기본값을 따라간다」는 설계 계약(회귀 테스트가 `== ConfluenceParams()`로 고정)을
깨뜨린다.

- **동결 스냅샷(고정함)**: 이중 필터(§2) wan133·wan142·wan152·wan154 + wan68·wan70·wan71·
  wan73·wan75·wan76·wan77·wan83·wan84·wan88·wan90·wan96·wan99·wan103·wan104·wan110·wan111·
  wan114·wan115·wan117·wan119·wan120·wan123(matched/fill)·wan126·wan134·wan137(resistance/
  phase2). (이 중 wan111/115/119/120도 밴드를 tap에 못 박은 동결 스냅샷이다.)
- **추적 모듈(고정 안 함 — 새 기본값 1.28을 따라간다)**: wan143 · wan145(entry) · wan149
  (merge/lifetime) · wan151(entry) · wan89(→ wan145/wan151의 null 팔이 `Arm.params()` 경유로
  물려받음). 이들은 밴드를 고정하지 않고 `adopted == ConfluenceParams()`를 주장하므로, 필터도
  1.28을 따라간다. **옛(필터 꺼진) CSV는 스냅샷이고, 새 기본값 위의 재측정은 별도 이슈**다
  (WAN-132→WAN-145, WAN-149→WAN-151과 같은 패턴 — OLD 모듈은 동결, NEW 모듈이 재측정).

**고정하지 않은 것(그 외)**: `wan155`·`wan161`(문턱을 자기 입력으로 명시) · A안 전용 리포트
(wan81/87/91 — A안은 필터를 안 읽고, close 팔은 명시적으로 끈다) · **채택 성과 wan95**(재산출
대상).

⚠️ **엣지 판정 계열이 또 한 번 얼어붙는다** — WAN-84/88/111/114/124/151은 전부 「필터 꺼진
시절」 기록이 된다. 새 기본값 위의 널·부품 분해는 **별도 이슈**다(WAN-132→WAN-145,
WAN-149→WAN-151과 같은 패턴).

## 5. 🚨 알고 받는 손해 (사용자가 알고 선택했다)

1. **낙관 체결 가정 위의 값이다** — 전부 `baseline`("지정가에 닿으면 무조건 체결"). WAN-154
   §4가 관통을 요구해도 `zone` 장벽은 두 TF 다 p=0.048을 지킴을 확인했지만, 실제 해소는 틱
   데이터(WAN-98, **Canceled**) 소관이다.
2. **승률 상승의 과반(55~63%)은 산수였다**(WAN-152) — 좁은 존이라 1R이 작아져 고정 1.5R 익절이
   가까워지는 기하 효과다. 손절 거리를 존폭에서 떼면 승률 격차의 과반이 사라진다.
3. **TRX 15m은 16거래로 사실상 매매가 멈춘다** — 손절폭 가드가 후보의 92.6%를 자른다.
4. **거래가 3분의 1로 준다**(15m 302 → 100). 앞으로의 모든 측정이 표본이 줄어 더 시끄러워진다.
5. **「엣지 없음」(WAN-84/88/111/114/124/151)이 뒤집힌 게 아니다** — 저 판정들은 *진입 규칙이
   무작위와 구분되나*를 물었고 존폭 축은 *이미 진입한 셋업 중 무엇을 버릴까*를 묻는다. 다른
   질문이다.
6. **문턱 1.28은 측정이 고른 값이 아니다** — 두 권고값(1.24·1.32)의 중간이고, 자유 파라미터를
   안 늘리려는 사용자 판단이다.
7. **엣지 판정 계열이 또 한 번 얼어붙는다**(§4).

### 📌 편중은 이 축에서 드물게 문제가 아니다

이 저장소의 플러스는 반복적으로 ETH 하나가 만들었는데(WAN-111/119/120/124/145/151), 존폭 축은
다르다 — 종목 하나씩 빼도 필터 팔 OOS는 네 셀 전부 플러스다(WAN-142/152/154가 각각 다른 자로
확인). 그래도 §5의 경고들은 그대로다.
