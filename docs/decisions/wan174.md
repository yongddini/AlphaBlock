# WAN-174 — 선물 웹소켓 0건의 진짜 원인: 엔드포인트 경로에 `/market`이 빠져 있었다

**판정: 차단이 아니었다.** ASTx도, 바이낸스의 클라우드 IP 차단도, 한국 리전 규제도 아니다.
수집기가 **옛 엔드포인트 경로**로 붙고 있었고, 그 경로는 **핸드셰이크만 성공시키고 데이터
프레임을 한 건도 보내지 않는다**. 경로에 `/market`을 넣자 로컬 맥에서 즉시 수신된다.

- 실행일 2026-07-24 · 로컬 맥(ASTx 상주 상태 그대로) · 공개 시세만 사용(읽기 전용)
- 수정: `data.stream.FUTURES_WS_BASE`, `dashboard.live_chart.LIVE_WS_BASE`

## 1. 실측 — 옛 경로 0건 · `/market` 경로 36건

같은 전송(`websockets` 라이브러리), 같은 호스트, 같은 심볼. **경로만** 다르다(각 20초):

| URL | 결과 |
| --- | --- |
| `wss://fstream.binance.com/stream?streams=btcusdt@kline_1m` (수집기 현행) | **0건** |
| `wss://fstream.binance.com/market/stream?streams=btcusdt@kline_1m` | **36건** |
| `wss://fstream.binance.com/market/ws/btcusdt@kline_1m` | **33건** |

⚠️ **0건 팔에서 예외도 연결 종료도 나지 않는다** — 101 Switching Protocols까지 정상이고
그대로 조용히 멈춘 것처럼 보인다. 이 **조용한 실패**가 오진의 원인이다.

## 2. 왜 그렇게 여러 번 오진됐나

원인 격리 매트릭스(엔드포인트 × URL 형식 × 압축, 각 12초):

| | `/stream?streams=` | `/ws/` |
| --- | --- | --- |
| **선물** `fstream` (옛 경로) | 0 / 0 | 0 / 0 |
| **현물** `stream.binance.com:9443` | 6 / 6 | 6 / 6 |

(각 칸 = 압축 `deflate` / `None` — **압축도 URL 형식도 원인이 아니다**.)

**현물은 접두사 없이 계속 동작하고 선물만 0건**이라, 관측이 언제나 "선물만 막힌다"로 나왔다.
그래서 지역 규제(바이낸스는 한국에서 파생 접근을 제한한다)나 클라우드 IP 차단 같은
**그럴듯한 설명에 계속 들어맞았다**. 실제로는 현물 경로가 옛 형식을 아직 받아 주고 선물
경로만 이전한 것뿐이다.

- **ASTx 원죄론이 틀린 이유**: ASTx가 상주하는 **바로 이 맥에서** `/market` 경로가 받는다.
- **클라우드 IP 차단론이 틀린 이유**: 사용자가 도쿄 Lightsail에서 `ccxt.pro`로 받았다.
  ccxt는 `wss://fstream.binance.com/market/ws/0`으로 붙는다 — **접두사를 이미 쓰고 있었다.**
  "ccxt는 되고 raw는 안 된다"는 대조의 정체가 라이브러리 차이가 아니라 **경로 차이**였다.
- **`SUBSCRIBE` 프레임 방식도 답이 아니었다** — 옛 경로에서 `{"method":"SUBSCRIBE",...}`를
  보내면 `{"result":null,"id":1}`로 **정상 ack까지 받고** 그 뒤 데이터가 안 온다.

## 3. 종단 검증 — 수집기 함수 그대로

`data.stream.stream_klines`를 기본 베이스로 40초(BTC·ETH · 1m · 인메모리 저장):

```
베이스 = wss://fstream.binance.com/market
하트비트(수신 메시지) : 103
파싱된 봉            : 103
마지막 봉            : ETH/USDT:USDT 1m close=1882.19 closed=False
```

파싱·심볼 매핑·소비 루프는 **한 줄도 안 고쳤다** — 결합 스트림 봉투
(`{"stream":..,"data":{"e":"kline",..}}`)가 그대로 오므로 `parse_kline_message`가 바로 받는다.

## 4. 이슈에 주는 함의 (⚠️ 사용자 결정 사항)

**이 이슈의 전제가 무너졌다.** WAN-174는 *"로컬 맥에서 수집이 불가하니 서버로 옮긴다"*가
동기였는데, **로컬 맥에서 수집이 된다.** 완료 기준의 핵심(«웹소켓이 1분봉을 실시간으로
받는다 — 로컬에서 막히던 것»)은 **서버 없이 충족됐다**(§3).

- **서버 이전이 여전히 필요한지는 사용자 결정이다.** 데이터 도달 문제로는 이제 필요 없다.
  남는 정당성은 **상시 가동**(맥은 잠자고 재부팅된다) 하나이고, 그건 원래 동기와 다른 문제다.
- **1단 배포 계층(systemd·셋업·런북)은 버리지 않는다** — 집 미니PC든 VPS든 상시 가동을
  택하면 그대로 쓴다.
- **서버 측 완료 기준(며칠 uptime · 재부팅 자동 복구)은 미검증이다** — 이 저장소에서
  검증할 수 없다(하드웨어 없음).

## 5. 안 한 것 · 범위 밖

- **`ccxt.pro`로의 전송 교체는 하지 않았다.** PM 코멘트가 후보로 적었지만, 원인이 경로로
  밝혀져 **불필요**하다. 교체는 파싱·재접속·테스트를 전부 다시 쓰는 큰 변경인데 얻는 게 없다.
- 기본값·토대·전략 로직 **불변**(인프라 전용). 실거래 보류
  `ALPHABLOCK_LIVE_TRADING=false` **유지**. 페이퍼 한정.
- 스트림 워치독(WAN-173)은 별개다 — ⚠️ 다만 이 사건은 워치독의 **필요성을 키운다**:
  "붙어 있는데 데이터가 0건"은 예외로 안 잡히므로 **수신 기반** 감시가 있어야 잡힌다.
