## 요약 — 결론 먼저

1. **재탭 RSI 게이트를 뺀다** — `ConfluenceParams.rsi_gate_mode` 기본값이
   `"first_tap_free"` → **`"unconditional"`**(게이트 자체가 없음). 사용자 결정: *"그래
   빼자."* 근거는 [WAN-114](wan114.md) ablation의 `L0r→L1`(게이트만의 순수 기여)이 **12셀
   중 8셀 음수 · 1h는 6셀 전부 음수**라는 것이다(§2).
2. **⚠️ 이건 "게이트를 껐다"이지 "RSI를 지웠다"가 아니다.** `rsi_length`·`rsi_mode`·
   `rsi_oversold`는 그대로 있고 진단·대시보드에 계속 쓰인다. 되돌릴 여지도 남겼다 —
   `rsi_gate_mode="first_tap_free"`로 옵트인하면 옛 규칙이 그대로 돈다(§1).
3. **⚠️ `"none"`이 아니라 `"unconditional"`인 이유가 이 이슈의 유일한 설계 판단이다.**
   이슈 본문은 `"none"`을 지정했지만 그 모드는 게이트 **판정**만 없앨 뿐 호출부의 워밍업
   요구(`rsi is not None`)를 남겨 **RSI 워밍업 구간의 탭을 계속 막는다** — 즉 "게이트를 뺐다"
   고 믿으면서 워밍업 필터만 남는다. 이슈가 스스로 적은 *"첫 탭 면제는 어차피 상위집합이라
   무의미해짐"*은 `"none"`에서는 **거짓**이고 `"unconditional"`에서만 참이다(§3).
4. **⚠️ 「엣지 없음」 판정(WAN-84/88/111)은 이 결정으로 바뀌지 않는다.** 게이트 제거는
   **드래그를 뗀 것**이지 신호를 더한 게 아니다. 그 재확인(매칭 널·체결 보수화 재검)은
   아직 **안 돌았다** — 이 PR은 4단 중 1단이다(§5).
5. **파급: 거래 집합이 넓어지므로 채택 수치가 전부 이동한다.** 게이트가 켜진 수치를 결론
   문장에 박아 둔 리포트는 `rsi_gate_mode="first_tap_free"`로 **명시 고정**해 당시 엔진의
   기록으로 보존했고(§4-1), 채택 성과(wan95)만 재산출했다(§4-3). WAN-112가 오프셋에 한 것과
   같은 패턴이되 **이동 폭이 다르다** — 오프셋은 같은 거래의 가격을 옮겼지만 게이트는
   **표본 자체를 바꾼다**(재산출된 wan95 기준 지정가 거래 **+19~27%**, 체결률 **51% → 81%**).
   ⚠️ WAN-114가 잰 "거래 13~14% 감소"는 **창을 못 박은 6심볼 15m/1h 격자**의 값이라 이
   수치와 직접 비교되지 않는다(창·심볼·TF가 다르다).

## 1. 무엇을 바꿨나

`ConfluenceParams.rsi_gate_mode` 기본값 **`"first_tap_free"` → `"unconditional"`**. 그 외
전략 필드는 하나도 건드리지 않았다(`retap_mode="every_tap"` 유지 — 재탭은 **계속 평가한다**,
다만 그 탭에 RSI 조건이 안 걸릴 뿐이다).

바뀐 규칙을 말로 쓰면:

| | WAN-81~122 (`first_tap_free`) | WAN-123~ (`unconditional`) |
| -- | -- | -- |
| 첫 탭(`tap_index=0`) | RSI 무관 무조건 진입(워밍업 NaN 포함) | 같음 |
| 재탭(`tap_index>=1`) | 롱 `RSI<=30` / 숏 `RSI>=70` 요구 | **무조건 진입** |
| 미충족 탭 | 기각하되 존은 소각 안 함 | (해당 없음 — 미충족이 없다) |

**되돌릴 여지**: 게이트 경로 코드는 지우지 않았다. `rsi_gate_mode="first_tap_free"`(또는
`"extreme"`/`"neutral"`)로 옵트인하면 그대로 돈다 — WAN-87이 `short_enabled=True` 경로를
남겨 둔 것과 같은 원칙이다.

## 2. 근거 — 게이트는 거래를 쳐내는데 그 쳐냄이 순손해다

[WAN-114](wan114.md) ablation이 `L0r`(재탭 on · 게이트 off) → `L1`(재탭 RSI 게이트 on)
단으로 **게이트만의 순수 기여**를 격리했다. 공식 렌즈(`baseline`) 기준 증분(%p):

| TF·구간 | `baseline` | `pen_5bp` | `pen_5bp_d50` |
| -- | -- | -- | -- |
| 15m IS | +2.78 | +3.66 | +6.34 |
| 15m OOS | **−2.76** | −1.18 | +0.37 |
| 1h IS | −2.26 | −1.12 | −1.74 |
| 1h OOS | **−2.28** | −2.28 | −2.29 |

읽는 법:

- **12셀 중 8셀이 음수**이고 **1h는 6셀 전부 음수**다.
- 유일하게 일관된 플러스는 **15m IS**(학습 구간)인데, 그것이 **OOS에서 −2.76%p로
  뒤집힌다**. 이 저장소가 반복해서 관찰한 IS→OOS 순위 뒤집힘이 **부품 축에서 재현된 것**이다
  (CLAUDE.md의 심볼 축·TF 축 뒤집힘과 같은 성질).
- 게이트는 거래를 **13~14% 쳐낸다**. 쳐내는 것 자체는 비용이 아니다 — 문제는 **그 쳐냄이
  마이너스**라는 것이다. 즉 게이트가 "고르는 방향"이 틀렸다.

**⚠️ 이것을 「RSI가 쓸모없다」로 일반화하지 말 것.** 이 표가 재는 것은 **극단 RSI를 재탭의
진입 조건으로 쓰는 것**이지 RSI 지표 자체가 아니다. 다른 방향(예: 게이트를 반대로 걸기)이나
다른 임계값은 이 표가 재지 않았다 — 그건 새 파라미터 탐색이고, 「엣지 없음」 판정 위에서
그 탐색을 시작하는 것은 순서가 틀렸다.

## 3. ⚠️ `"none"`이 아니라 `"unconditional"`인 이유 (설계 판단)

이슈 본문은 `rsi_gate_mode="none"`을 지정했다. **그대로 넣었으면 게이트가 절반만 꺼졌다.**

두 진입 경로가 **모드와 무관하게** 워밍업을 먼저 요구하기 때문이다:

```python
# A안 strategy/confluence.py · B안 backtest/substep.py — 같은 모양
confirmed = rsi is not None and rsi_gate_passes(rsi, mode=...)
#           ^^^^^^^^^^^^^^^^ 모드가 "none"이어도 이 조건은 남는다
```

`rsi_gate_passes(mode="none")`은 항상 True지만 **앞의 `rsi is not None`이 워밍업 구간(첫
14봉)의 탭을 막는다**. 그래서:

- `"none"` = 게이트 **판정**은 없다 + **워밍업은 여전히 요구**한다.
- `"unconditional"` = 게이트가 **아예 없다**(RSI를 읽지도 않는다).

이 차이는 이미 관측된 적이 있다 — [WAN-114 §`L0r`](wan114.md)이 *"`L0r`은 `L0`의 깨끗한
상위집합이 아니다"*라고 적어 둔 비대칭이 정확히 이것이다. 그리고 이슈 본문이 게이트 제거를
정당화하며 쓴 문장 *"첫 탭 면제는 어차피 상위집합이라 무의미해짐"*은 **`"none"`에서는
성립하지 않는다**(워밍업 첫 탭을 `first_tap_free`는 넣고 `none`은 막는다). `"unconditional"`
에서만 그 문장이 참이 되므로, 이슈의 **의도**를 구현한 것이 `"unconditional"`이다.

**`"none"`의 의미는 바꾸지 않았다** — WAN-114 `L0r`이 그 뜻으로 측정돼 CSV에 고정돼 있고,
의미를 갈아치우면 그 표가 조용히 다른 것을 재게 된다.

이 부류의 사고는 이 저장소에서 세 번 있었다: `funding_enabled`만 켜고 요율을 안 넘긴
WAN-91, `entry_mode`가 라벨일 뿐이던 WAN-95, 그리고 2bp를 `0.0002`로 넣을 뻔한 WAN-112.
**전부 "설정은 바꿨는데 동작은 그대로"였다.** 그래서 이 이슈는 라벨이 아니라 **동작**을
검사하는 회귀 테스트를 남긴다:

- `test_wan123_default_gate_is_off_for_retaps` — 재탭 + RSI 50인데 진입하는가(A안).
- `test_unconditional_fills_retap_despite_failing_rsi_gate` — 같은 것(B안 시뮬레이터).
- `test_wan123_default_gate_enters_on_warmup_retap` / `test_unconditional_fills_during_warmup_but_none_does_not`
  — **`unconditional`과 `none`이 워밍업에서 갈리는지**. 이게 §3의 핵심을 못 박는다.
- `test_wan123_default_is_a_superset_of_first_tap_free` — 이슈의 「상위집합」 주장이 참인가.
- `test_default_gate_is_off_and_legacy_pin_differs` — 핀이 기본값과 **다른지**(같아지면
  아래 §4의 고정이 전부 조용한 no-op이 된다).

## 4. 파급 — 무엇이 고정됐고 무엇이 움직였나

게이트 제거는 **거래 집합 자체를 13~14% 넓힌다**. 오프셋(WAN-112)이 같은 거래의 가격을 조금
옮긴 것과 **성격이 다르다** — 표본이 바뀌므로 옛 표의 건수·승률·수익이 재현되지 않는다.

### 4-1. 옛 엔진으로 **명시 고정**한 리포트

전부 `rsi_gate_mode="first_tap_free"`를 박았다(`backtest.harness.LEGACY_RSI_GATE_MODE`).
공통 이유: **결론 문장이 게이트가 켜진 수치에 박혀 있어**, 기본값을 따라 움직이면 본문과
어긋난다.

| 리포트 | 고정한 이유 |
| -- | -- |
| `wan81_engine_replacement_report` | "WAN-81 신 엔진" 정의에 **재탭 게이트가 포함**된다 |
| `wan83_position_conflict_report` | 첫 탭 소각 계측 — 게이트가 있어야 "재탭이 막힌다"가 성립 |
| `wan84_new_engine_validation` | 「엣지 없음」 판정(숏 포함 정의 보존) |
| `wan87_long_only_report` | 롱/숏 대조 |
| `wan88_long_only_validation` | 「유효 16셀 중 유의 0개」 |
| `wan91_funding_cost_report` | 펀딩 on/off 델타(거래 집합 고정이 전제) |
| `wan96_fill_conservatism_report` | 거래-수익 비대칭("거래 4.7%↓인데 수익은 사라진다") |
| `wan99_zone_limit_offset_report` | 오프셋 0 행이 WAN-95/96과 맞물리는 검산 |
| `wan103` / `wan110` | 단일 vs 다중 비교(두 항이 같은 풀이어야 한다) |
| `wan104_offset_incremental_report` | 증분 분해("13건 vs 917건") |
| `wan111_symbol_universe_report` | 6심볼 판정 + **WAN-114 `L2`와의 비트 단위 검산** |

⚠️ **`wan111`은 스스로 "기본값을 따라가라"고 적어 둔 리포트였다**(오프셋 하드코딩 금지).
그 원칙이 오프셋에는 맞고 게이트에는 안 맞는 이유는 위의 **이동 크기** 차이다. 게다가
WAN-114 사다리(`L2` 행)가 그 CSV와 비트 단위로 맞물리는 것이 두 리포트의 상호 검산인데,
사다리는 게이트를 고정하므로 wan111만 따라가면 그 검산이 깨진다.

### 4-2. ablation 사다리 — CSV는 안전하고 **주장 하나가 개정된다**

`wan114`·`wan115`·`wan119`·`wan120`은 **모든 단(rung)이 `rsi_gate_mode`를 이미 명시**하고
있어 CSV가 비트 단위로 재현된다. 다만 그 표들이 달고 있던 **"`L2` == 채택 기본값"이라는
라벨은 이제 거짓**이다 — 정확히는 **"`L2` == WAN-122까지의 채택 기본값"**이다.

고정이 맞는 선택인 이유: 이 사다리들의 존재 이유가 **부품별 증분 분해**인데, 게이트를
기본값에 맡기면 `L0r→L1`(게이트의 기여)이라는 단 자체가 사라지고 `L1→L2` 증분도 "볼린저의
기여"가 아니게 된다. 특히 **WAN-114는 이 결정을 낳은 근거 그 자체**라, 결정 이후의 엔진으로
다시 도는 것은 의미가 없다.

### 4-3. 재산출한 리포트

**`wan95_zone_limit_report`(채택 성과)** — 이 리포트의 역할이 "**지금 채택된 것**의 성과"라
고정하지 않고 다시 돌렸다. 두 팔(지정가·종가)이 **함께** 게이트를 잃으므로 이 표가 격리하는
변수(진입 방식)는 그대로다. `zone_limit` 행, TF별 3심볼 평균:

| TF | `total_return` | 거래 수 | 체결률 | 승률 |
| -- | -- | -- | -- | -- |
| 15m | +67.04% → **+83.47%** | 2426 → 2957 (+21.9%) | 51.2% → **81.6%** | 50.46% → 50.49% |
| 1h | +35.91% → **+41.06%** | 692 → 840 (+21.4%) | 50.3% → **78.0%** | 51.14% → 50.71% |
| 4h | +12.81% → **+17.74%** | 174 → 207 (+19.0%) | 57.4% → **79.9%** | 51.82% → 52.74% |
| 1d | +1.01% → **+1.20%** | 30 → 32 (+6.7%) | 57.9% → **71.4%** | 44.37% → 44.44% |

**📌 체결률 ~51% → ~81%가 이 재산출의 가장 큰 사실이고, WAN-114가 이미 예고한 값이다.**
그쪽 분해는 *"체결률 ~51%는 시장이 아니라 규칙 층이 만든 것이다 — 존-단독은 **체결률
100%**이고, RSI 게이트가 100%→62%, 볼린저 재산정이 62%→51%로 깎는다"*라고 적었다. 게이트를
빼자 그 몫이 돌아왔다 — **두 리포트가 서로를 검산한다**(게이트가 정말 꺼졌다는 독립 증거이기도
하다). CLAUDE.md·WAN-96이 "체결률 ~50%가 낙관이라 걱정"이라 한 그 수치의 절반은 **시장이
아니라 우리 규칙**이었다는 뜻이다. ⚠️ 단 **큐 우선순위 걱정 자체는 그대로다** — 81%도
"닿으면 체결"(`baseline`) 가정 위의 값이다.

⚠️ **위 표의 좌변(옛 수치)과의 차이를 「게이트 효과」로 읽지 말 것.** `--years 3` 창은 마지막
봉 기준이라 실행 시각이 다르면 창이 미끄러진다(CLAUDE.md) — 좌변은 WAN-112 실행(2026-07-15),
우변은 이 실행(2026-07-16)이라 **창이 하루 다르고**, 그 차이가 델타에 섞여 있다. 게이트만의
격리된 기여는 **창을 못 박은** WAN-114 사다리(`L0r→L1`, §2 표)가 낸다. 여기 표는 "게이트가
얼마를 벌었나"가 아니라 **"지금 채택된 것의 성과가 무엇인가"** 를 읽는 표다.

⚠️ **수익이 다 오른 것도 아니다** — 셀 단위로는 `SOL 1h`(+44.15% → +34.00%) · `SOL 4h`
(+18.60% → +16.11%) · `SOL 1d`가 내려갔다. 3심볼 평균이 오른 것과 별개다.

`tests/test_run_regression_real_data.py::test_cli_defaults_reproduce_wan95_zone_limit_cell`이
"CLI 기본 실행 == WAN-95 셀"을 실데이터로 고정하므로(BTC 1h), 이 재산출이 곧 그 계약의
갱신이다 — 그 테스트가 통과한다는 것은 **기본값 경로와 리포트가 같은 엔진을 돈다**는 증거다.
