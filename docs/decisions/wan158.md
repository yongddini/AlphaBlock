## 요약 — 결론 먼저

1. **파라미터를 만들되 기본값은 바꾸지 않는다.** `ConfluenceParams.max_zone_width_atr`
   (기본 `None` = 꺼짐 = 전부 매매)와 CLI 축 `--max-zone-width-atr`가 생겼다. 채택(A)이
   아니므로 **옛 리포트 재산출도 `LEGACY_*` 고정도 없다** — 기본 실행은 예전과 **비트
   단위로 같다**(§2).
2. **왜 A가 아닌가** — 측정은 세 관문을 통과했지만 (a) 전부 **낙관 체결 가정**(`baseline`)
   위의 값이고, (b) 승률 상승의 **과반이 기하**(익절선이 가까워진 산수)였으며, (c)
   **실시간에서 문턱을 어떻게 운용할지**가 안 정해져 있었다. **먼저 켜 보고 판단한다**(§1).
3. **문턱은 분위가 아니라 절대값으로 노출한다** — 이것이 이 이슈의 핵심 설계 결정이다.
   측정 모듈은 문턱을 "IS 공통 풀의 하위 1/3 분위"로 잡았는데 **실시간에는 그런 풀이
   없다**. 다행히 그 분위가 종목·TF를 가리지 않고 **1.16~1.39에 몰려** 있어 단일 절대
   문턱으로 근사된다(§3).
4. **🚨 「켜면 돈을 더 번다」로 인용 금지** — 이 배선은 **측정을 재현하는 통로**이지 새
   수치가 아니다. 「엣지 없음」(WAN-84/88/111/114/124/151)은 그대로이고, TRX는 15m에서
   매매가 거의 사라진다(§4).
5. **범위 밖을 명시한다** — 손절폭 가드(WAN-76/79) · 실시간 러너 배선(WAN-45) ·
   대시보드 폼(WAN-102) · 새 격자는 **손대지 않았다**(§5).

## 1. 왜 B(옵트인)인가 — A와 C를 왜 안 골랐나

관문 세 개는 **실제로 통과했다**:

| 이슈 | 물은 것 | 답 |
| -- | -- | -- |
| [WAN-142](wan142.md) | MDD 개선이 「덜 베팅해서」인가 | **아니다** — 매칭 대조군 네 셀 전부 p=0.048(하한) |
| [WAN-152](wan152.md) | 「익절선이 가까워져서」인가 | **그것만은 아니다** — ATR 장벽에서도 필터만 플러스 |
| [WAN-154](wan154.md) | 관통 요구·문턱을 바꿔도 버티나 | **버틴다** — `zone` 축은 두 TF 관통 생존 · 문턱 3종 전부 유의 |

그런데도 기본값을 안 옮긴 이유는 셋이다.

1. **전부 `baseline`(닿으면 체결) 위의 값이다.** WAN-154 §4가 관통을 요구해도 `zone`
   장벽은 두 TF 다 유의성을 지킨다고 확인했지만(통제용 `atr`·`zone_height` 장벽의 15m은
   **소멸**), 낙관 렌즈 자체의 해소는 틱·호가([WAN-98](https://linear.app/wannabe-millionaire/issue/WAN-98))
   소관이고 **그 이슈는 Canceled다**. 그것을 기다리는 C(보류)는 **무기한 대기**라 고르지
   않았다.
2. **승률 상승의 과반은 산수였다.** WAN-152: 손절 거리를 존폭에서 떼면 (필터 − 매칭)
   승률 격차의 **55~63%가 사라진다**. 판정은 그 나머지와 수익 우위로 냈다 — 「좁은 존을
   고르는 눈」의 몫은 처음 보였던 것보다 작다.
3. **실시간 운용 방법이 안 정해져 있었다.** 측정의 문턱은 백테스트에서만 계산 가능한
   분위였다(§3). 그 상태로 기본값을 옮기면 실거래 배선에서 다시 막힌다 — WAN-132가
   `tap` 밴드에서 겪은 그 부류다.

**D(페이퍼 검증 먼저)** 는 [WAN-45](https://linear.app/wannabe-millionaire/issue/WAN-45)
(라이브 러너 ↔ LimitOrderBook 배선)가 아직 안 됐으므로 지금 고를 수 있는 선택지가 아니었다.
B는 그 사이에 **백테스트에서 자유롭게 켜 보는** 길을 연다.

## 2. 무엇을 만들었나

| 층 | 추가 | 기본값 |
| -- | -- | -- |
| 파라미터 | `ConfluenceParams.max_zone_width_atr` (ATR 배수) | `None` = 꺼짐 |
| | `ConfluenceParams.zone_width_atr_length` | `14`(측정 모듈과 같은 길이) |
| 판정 | `ConfluenceParams.zone_width_filter_passes(ob, atr)` | 꺼져 있으면 항상 통과 |
| 엔진 | `build_zone_limit_candidates`가 **지정가 확정 이후** 후보를 거른다 | — |
| 조립 | `harness.build_params(max_zone_width_atr=...)` · `RunRow.max_zone_width_atr` | `None` |
| CLI | `--max-zone-width-atr none,1.24` (콤마 = 격자 축) | `(None,)` |

**판정 규칙**: `(존폭 top−bottom) ÷ ATR ≤ 문턱`이면 진입, 아니면 **주문을 걸지 않는다**.
ATR은 **탭 봉 직전 확정봉**에서 읽는다 — 탭 봉 자신의 ATR은 그 봉 종가를 알아야 나오므로
룩어헤드다(WAN-115/119/132가 밴드에서 겪은 그 자리). 측정 모듈
(`wan117._FeatureExtractor.features_for`)의 `zone_width_atr` 특징과 **같은 정의**다.

**끄면 예전과 같다** — `None`이면 엔진이 ATR을 **계산조차 하지 않으므로** 기존 CSV가 비트
단위로 재현된다. 회귀 테스트(`tests/test_wan158_zone_width_filter.py`)가 라벨이 아니라
**동작**으로 고정한다: (a) 꺼짐일 때 후보 집합·`ZoneLimitStats` 동일, (b) 켜면 같은 셋업이
문턱 하나로 **실제로 사라진다**, (c) 문턱을 두 봉의 ATR **사이**에 놓아 엔진이 직전 확정봉을
읽는지 가른다, (d) 워밍업(NaN)·비양수 ATR은 **통과가 아니라 기각**이다.

⚠️ **왜 `ConfluenceParams`이고 `OrderBlockParams`가 아닌가** — 존폭 필터는 존을 **덜
탐지**하는 규칙이 아니라 탐지된 존 중 **진입 후보를 거르는** 규칙이다(측정 모듈도 후보
리스트를 걸렀다). 탐지 파라미터에 넣으면 `--combine-obs`처럼 값마다 오더블록을 다시
탐지하게 되고 아카이브·대시보드 표시까지 바뀐다 — 이 결정의 범위가 아니다.

⚠️ **종가 진입(A안)과 함께 쓰면 거부한다** — A안(`entry_mode="close"`)은 이 필드를 읽지
않으므로 `evaluate()` · `build_params` · `Grid`가 각각 `ValueError`를 낸다. 조용히 무시하면
"필터를 켜고 돌렸다"고 믿는 표에 라벨만 붙는다(WAN-95의 교훈).

## 3. 문턱 — 분위가 아니라 절대값

측정 모듈(`wan152`·`wan154`)은 문턱을 **IS 공통 풀의 하위 1/3 분위**로 잡았다. 그 값을
그대로 파라미터로 옮기면 백테스트에서만 계산 가능한 수가 된다. `wan154_threshold_stability.csv`
집계 결과, 하위 1/3 분위가 **종목·TF를 가리지 않고 좁은 띠에 몰려 있다**:

| TF | IS 문턱 범위 | 평균(권고) | OOS에서 실제로 거른 비율 |
| -- | -- | --: | -- |
| 15m | 1.155 (SOL) ~ 1.276 (BNB) | **1.24** | 27.9 ~ 36.5% |
| 1h | 1.225 (SOL) ~ 1.390 (ETH) | **1.32** | 28.2 ~ 53.4% |

→ 그래서 파라미터는 **단일 절대 문턱**(`max_zone_width_atr`)이다. 종목별 분위를 다시 잡지
않고도 백테스트 판정을 근사한다.

⚠️ **단위는 ATR 배수지 퍼센트가 아니다.** `1.24`는 "존폭이 ATR의 1.24배 이하"라는 뜻이고,
퍼센트로 착각한 `0.0124`를 넣으면 **거의 모든 셋업이 걸러진다**(WAN-112가 `2.0`을 `0.0002`로
넣어 무효과를 겪은 그 함정의 반대 방향). 회귀 테스트가 이 단위를 동작으로 남긴다.

⚠️ **TRX 1h는 같은 문턱에 53.4%가 걸린다** — 종목별 편차가 커서 "하위 1/3"이 실제로는
"하위 절반"이 되는 종목이 있다. 절대 문턱을 쓰는 대가다.

## 4. 켜기 전에 알아야 할 것 (전부 선행 리포트에 있다)

1. **낙관 가정 위의 값이다** — `baseline`("닿으면 체결"). 실제 해소는 틱·호가(WAN-98,
   Canceled) 소관.
2. **승률 상승의 과반은 기하였다** — WAN-152: 장벽을 존폭에서 떼면 승률 격차의 55~63%가
   사라진다.
3. **🚨 TRX는 15m에서 매매가 거의 사라진다** — `zone` 장벽 15m에서 필터 후보의 **92.6%를
   손절폭 가드(`min_stop_distance_fraction`)가 잘라 16거래**로 무너진다(WAN-142/154). 켜면
   **종목별 거래량이 극단적으로 갈린다**. ⚠️ 가드 변경은 **WAN-76/79 소관이고
   재-베이스라인**이라 이 이슈에서 손대지 않았다 — 현상만 기록한다.
4. **거래가 3분의 1로 준다** — 15m OOS 302 → 100건(WAN-158 이슈 본문 표). 표본이 줄어
   앞으로의 모든 측정이 더 시끄러워진다.
5. **「엣지 없음」(WAN-84/88/111/114/124/151)이 뒤집힌 게 아니다** — 저 판정들은 *「진입
   규칙이 무작위와 구분되는가」*를 물었고 존폭 축은 *「이미 진입한 셋업 중 무엇을 버릴
   것인가」*를 묻는다. **다른 질문이다.**

* ❌ **새 격자·새 측정** — WAN-142/152/154에서 끝났다. 이 결정은 **새 숫자를 내지 않았다**.

`combine_obs=False`. **실거래 보류 유지**(`ALPHABLOCK_LIVE_TRADING=false`).

## 6. 사용법

```bash
# 1h에서 필터 on/off를 한 표에 (권고 문턱 1.32)
uv run python -m backtest.run --tf 1h --max-zone-width-atr none,1.32 --oos

# 문턱 민감도 — 하위 1/4 · 1/3 · 1/2 근방을 훑는다
uv run python -m backtest.run --tf 15m --max-zone-width-atr none,1.11,1.24,1.45 --oos

# 종가 진입(A안)과는 못 쓴다 — ValueError로 거부한다
uv run python -m backtest.run --entry-mode close --max-zone-width-atr 1.24   # ❌
```

표에 `zw_atr` 열이 붙고, 값이 하나뿐이면(= 기본 실행) 예전처럼 접힌다. 파이썬에서는
`ConfluenceParams(max_zone_width_atr=1.32)`로 직접 켠다.
