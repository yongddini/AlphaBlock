## §0 파일럿 설계 — 왜 「QC에서 바로」가 아니라 「로컬 감사 먼저」인가

이슈의 관건은 *"봉내 라이브 볼린저밴드(intrabar_live)를 QC에서 똑같이 재현하는 것 —
안 맞추면 비교 무의미"* 다. QC 클라우드에서 바로 돌려 보면 **데이터 축**(QC CoinAPI vs
우리 수집)과 **로직 축**(이벤트 구동 재배열이 정본과 같은가)이 동시에 움직여 불일치의
출처를 가를 수 없다(WAN-74 감사가 확립한 원칙). 그래서 파일럿을 두 층으로 갈랐다:

1. **로컬 감사(이번 PR, 완료)** — 데이터를 저장소로 고정하고 **로직 축만** 움직인다.
   QC의 `OnData` 실행 모델이 강제하는 재배열(셋업별 배치 → 시간 순서 단일 패스 + 온라인
   북)을 `qc/event_engine.py`로 만들고, 정본 엔진과 같은 데이터 위에서 거래가 일치하는지
   잰다(`backtest/wan181_qc_pilot.py`).
2. **QC 클라우드 감사(다음 단계, 사용자 QC 계정 필요)** — 로컬 감사를 통과한 드라이버를
   QC 데이터 위에서 재생(`qc/algorithm.py`)해 **데이터 축만** 움직인다.

**포팅 원칙(WAN-171의 「같은 회계 함수 공유」를 길 A에 적용)**: 규칙 원천(밴드
`RealtimeBand` · 가격 사슬 `IntrabarLiveLimit` · 사이징 `position_size` · 비용 `CostModel` ·
펀딩 `cumulative_funding_cost` · 탐지 `OrderBlockDetector`)은 **재구현하지 않고 그대로
import** 한다 — QC 클라우드에 저장소를 통째로 올리는 방식이라 가능하고, 두 벌을 만들면
어느 쪽이 맞는지 알 수 없게 된다(WAN-100의 교훈). 새로 쓴 것은 QC가 강제하는 **실행
모델의 재배열**뿐이고, 그 재배열이 감사 대상이다.

## §1 로컬 불일치 감사 결과 (BTC 1h × 못 박은 채택 창)

셀: `BTC/USDT:USDT` × `1h` × `2020-09-15 ~ 2026-07-22`(WAN-182 채택 창) × 채택 기본값
(`ConfluenceParams()` — 오프셋 2bp · `intrabar_live` · `unconditional` · 존폭 1.28 ·
고정 1.5R · 롱 온리). 재현: `uv run python -m backtest.wan181_qc_pilot`
(요약: `backtest/reports/wan181_qc_pilot_summary.md`).

### 결과 — 후보 축 완전 동치 · 잔여 간극은 시퀀싱 타이브레이크 하나로 특정됨

| 축 | 정본 | 이벤트 재배열 | 판정 |
| -- | --: | --: | -- |
| eligible 셋업 | 485 | 485 | ✅ 일치 |
| 체결(filled) | 344 | 344 | ✅ 일치 |
| 후보 진입·청산 7필드 | — | — | ✅ **344행 전부 비트 일치** |
| 같은 분 체결 경합(타이) | — | 19회 | ⚠️ 아래 참고 |
| 최종 거래 수 | 254 | 254 | ✅ 일치 |
| 승률 | 61.02% | 61.02% | ✅ 일치 |
| 총수익률 | +98.94% | +99.86% | ⚠️ +0.92%p |
| MDD | 5.09% | 5.09% | ✅ 일치 |
| 불일치 거래 | — | 32건 | ⚠️ 전부 타이의 몫 |

- 📌 **밴드·체결·청산의 재배열은 정확하다** — 이슈가 관건으로 지목한 봉내 라이브
  볼린저(intrabar_live)를 포함해, 후보(셋업) 축은 **344건 × 7필드 전부 비트 일치**다.
  eligible/filled/penetrations 통계도 일치한다.
- 📌 **잔여 간극의 출처가 하나로 특정됐다** — 정본 시퀀서는 같은 분에 체결이 경합하면
  `(entry_time, exit_time)` 정렬로 **청산이 이른 쪽**을 고르는데, 청산 시각은 그 시점의
  **미래 정보**라 실거래도 QC도 어떤 온라인 집행도 재현할 수 없다. 6년 창에서 그 경합이
  19회 있었고, 슬롯 주인이 갈리면 이후 배치가 연쇄로 밀려 최종 거래 32건이 달라졌다.
  크기는 총수익 +0.92%p(거래 수·승률·MDD는 동일)다.
- ⚠️ **"이벤트 쪽이 더 번다"로 읽지 말 것** — 타이브레이크는 방향성 없는 동전 던지기라
  이 +0.92%p는 이 셀의 우연이다. 읽을 것은 **간극의 크기가 이 수준**이고 출처가 백테스트
  쪽의 재현 불가능한 관행이라는 사실이다. 뒤집어 말하면 **정본 백테스트 수치에는 온라인
  집행이 원리적으로 재현할 수 없는 타이브레이크 성분이 이만큼 들어 있다** — QC(또는 어떤
  러너)와 우리 표를 대조할 때 이 성분은 불일치가 아니라 바닥 잡음으로 읽어야 한다.

대조 축 3개:

- **후보(셋업) 축** — 정본 `build_zone_limit_candidates` vs 이벤트 엔진 가상 후보를
  `(trigger_time, tap_index, zone_key)`로 조인, 진입·청산 7필드 대조. 밴드 재산정·체결·
  청산 산출의 재배열 검증.
- **시퀀싱 축** — 정본 시퀀서는 같은 분 체결 경합을 **청산 시각**(그 시점의 미래 정보)으로
  깨는데 온라인 북(실거래·QC가 실제로 할 수 있는 것)은 등록 순서로 깬다. 타이 발생
  횟수가 이 축의 위험 지표다(0이면 동치).
- **헤드라인** — 최종 거래·성과(거래 수·승률·수익·MDD) 대조.

합성 데이터 동작 고정: `tests/test_qc_event_engine.py`(후보·통계·거래 동치 + 범위 밖
파라미터 명시적 거부 + 시간 역행 거부).

## §2 두 길 교환비 표 (기록용 — 길은 이미 A로 결정됨)

| 축 | 길 A — QC 클라우드 | 길 B — 로컬/서버 |
| -- | -- | -- |
| 우회하는 것 | 로컬 맥 ASTx 웹소켓 차단(WAN-174) 통째로 우회 · 수집기 워치독(WAN-173) 페이퍼 트랙 불필요 · 24시간 서버 관리 소멸 | 없음(전부 직접 짊어짐) |
| 새로 드는 것 | 이벤트 재배열 층(`qc/` — **이번 파일럿로 로컬 검증 완료**) · QC 데이터/집행 축 감사(미실행) · QC 구독료(라이브/페이퍼 노드 유료, A/B 두 장부) | 서버 이전(WAN-174) · 수집기/워치독 운영 지속 |
| 엔진 연속성 | **부분 확보** — 규칙 원천을 재구현 없이 vendoring(§0 원칙)하고 재배열만 새로 씀. 로컬 감사가 그 재배열의 동치를 잼 | 완전(러너·백테스터가 같은 회계 함수, WAN-171 설계) |

| 독립 교차검증 | **보너스로 얻음** — QC 데이터·집행이 우리 수치를 재현하는지가 곧 조용한 버그 감사(이 저장소가 반복해서 데인 유형) | 없음(같은 엔진을 다시 돌릴 뿐) |
| 실시간 간극 | §4의 스트리밍 간극 2개(탭 확정 시점 · 물리 주문 vs 가상 스킵) — **처리 방안이 사용자 결정으로 남음** | 같은 간극이 존재하나 러너(WAN-45)가 이미 자기 방식으로 처리 중 |

## §3 WAN-174 / WAN-171 우선순위 정리 권고

- **WAN-174(리눅스 서버 이전)**: 페이퍼 목적의 긴급성은 **소멸**(QC가 데이터·실행을
  클라우드에서 담당). 단 이슈 본문이 적시한 대로 수집기는 대시보드·백테스트용 신선
  데이터에도 쓰이므로 **완전 폐기가 아니라 우선순위 강등**을 권고한다 — 다음 격자
  실험 전에 9종목 수집이 밀리면 창이 어긋난다(WAN-111 경고). 강등 폭은 사용자 판단.
- **WAN-171(레버리지 북 페이퍼 배선)**: 길 B 전제의 러너 배선 부분은 **범위 축소**를
  권고한다. 단 **레버리지 북 회계 로직 자체(WAN-169/180)는 QC 재구현(②단)의 입력**이라
  살아 있다 — "칸=(종목,TF) 1포지션 · 공유 자본 · N배/cap-only"를 QC 포팅층
  (`qc/event_engine.py`의 다음 단계)으로 옮기는 작업으로 성격이 바뀐다.
- 이 정리는 **권고**다 — 이슈 상태 변경(Todo 강등·범위 수정)은 PM/사용자 몫이다.

## §4 남은 축 · 다음 단계 (QC 페이퍼는 포팅의 하류지 지름길이 아니다)

1. **데이터 축 감사(다음, 사용자 행동 필요)** — `qc/algorithm.py`를 QC 클라우드
   백테스트로 실행(사용자 QC 계정·프로젝트 업로드 필요 — 개발자가 계정을 만들지
   않는다). ObjectStore가 남기는 `wan181_qc_trades.csv`를 로컬 `--no-funding` 실행의
   `wan181_qc_pilot_trades_event.csv`와 diff — 로직이 동일하므로 차이는 순수하게
   **QC CoinAPI vs 우리 수집 데이터**의 몫이다.
   - ⚠️ 펀딩: 1단은 QC에서 펀딩 시계열을 얻지 않으므로 양쪽 다 펀딩 끄고 대조한다.
     QC 페이퍼(3단)에서의 펀딩 처리(QC 계정 회계 vs 우리 산식)는 2단에서 설계한다.
   - ⚠️ 의존성: QC 클라우드 파이썬에 `pandas`·`pydantic`이 필요하다(지원 목록 기준
     가능 — 막히면 `strategy/models.py`의 pydantic 의존을 얇은 dataclass 셔로 바꾸는
     비용이 추가된다. 그때 별도 이슈).
2. **집행 축(2단)** — QC `LimitOrder` + 1분 스냅샷 체결 모델로 전환. **스트리밍 간극
   2개의 처리 방안이 선행 사용자 결정이다**:
   - (a) **탭 확정 시점**: 정본 셋업은 탭 봉 **슬롯 시작부터** 체결을 허용하지만(탭
     시그널이 그 봉에 있다는 것은 봉이 닫혀야 확정), 실시간 러너는 탭을 봉 진행 중에야
     감지한다. 오프셋 +2bp 지정가는 존 근단 **위**라 존 터치 전에 닿을 수 있어, 이
     간극이 0이 아니다.
   - (b) **물리 주문 vs 가상 스킵**: 정본(그리고 이 파일럿의 온라인 북)은 모든 셋업의
     체결을 **가상으로** 계산하고 북 점유 시 스킵한다. 실주문 하나만 거는 페이퍼는
     "북이 비었을 때만 주문을 낸다"가 되어 다른 거래 집합이 나온다. 선택지: 가상 추적 +
     시장가 미러링(정본 충실, 지정가 현실성 상실) vs 실지정가(현실 충실, 정본과 간극
     수용). **개발자 임의 착수 금지.**
3. **레버리지 북 QC 재구현(②단)** — WAN-169/180의 칸·공유 자본·cap-only 회계를 이벤트
   층으로. 그 뒤에야 ③ QC 페이퍼(A=결합 3배 / B=cap-only 5배 두 장부)다.
