# WAN-130: 채택 기본값에서의 재확인 — TF 순위 · 다중 대 단일 부호

**판정: 두 결론 모두 유지된다.** (1) 채택 TF 순위(15m > 1h > 4h, [WAN-107](wan107.md))와
(2) 다중 포지션의 **정반대 부호**(15m은 다중 우세 · 1h는 단일 우세, [WAN-103](wan103.md)
§A/§C)가 **오늘의 채택 기본값**에서 그대로 나온다. **기본값은 바꾸지 않았다**(확인 전용,
실거래 보류 `ALPHABLOCK_LIVE_TRADING=false` 유지).

**재현**

```
uv run python -m backtest.run --symbol BTCUSDT,ETHUSDT,SOLUSDT --tf 15m,1h,4h \
    --positions single,3 --oos --start 2023-07-15 --end 2026-07-15 --jobs auto \
    --format csv --out backtest/reports/wan130_adopted_defaults_confirmation.csv

# 표만 다시 접기(엔진 무관 집계)
uv run python scripts/wan130_summarize.py backtest/reports/wan130_adopted_defaults_confirmation.csv
```

원본: [`wan130_adopted_defaults_confirmation.csv`](../../backtest/reports/wan130_adopted_defaults_confirmation.csv)
(54행 = 3심볼 × 3TF × 3구간 × 2포지션 정책). 새 `wanNN_*.py` 리포트 모듈은 만들지 않았다
([WAN-101](https://linear.app/wannabe-millionaire/issue/WAN-101) 규약) — 숫자는 전부 범용
CLI가 냈고, 다중 포지션을 CLI에서 돌릴 수 있도록 **포지션 정책 축(`--positions`)** 만
추가했다(아래 §4).

## 배경 — 무엇이 열려 있었나

CLAUDE.md가 [WAN-112](wan112.md) 파급 문단에 직접 적어 둔 구멍이다: *"WAN-107(채택 TF)·
WAN-108/110(다중 포지션) 판정 수치는 아직 재산출 안 했다."* 그 뒤 두 개가 더 얹혔다 —
[WAN-123](wan123.md)이 RSI 재탭 게이트를 빼고([WAN-116] 결정 B), [WAN-132](wan132.md)가
진입가 정본을 `tap` → `intrabar_live`로 옮기면서 **세 판정의 리포트 모듈을 옛 값으로 명시
고정**했다. 즉 지금 인용되는 TF·다중 포지션 판정은 **오프셋 0bp · 옛 밴드 · 게이트 on**
위에 서 있었고, 채택 기본값에서 검증된 적이 없었다.

이 문서는 **재산출이 아니라 확인**이다(이슈 §"먼저 결정해야 할 것"): 옛 md/csv는 손대지
않고(비트 단위 재현 보존), **결론을 가르는 셀만** 오늘의 채택 기본값에서 다시 내
**부호와 순위**를 본다.

## §0 — 이 표가 도는 엔진 (옛 표와 무엇이 다른가)

| 축 | WAN-107/108/110 표 | **이 확인 격자** |
| -- | -- | -- |
| 지정가 오프셋 | 0bp | **2bp**(WAN-112) |
| 밴드 표본 | `tap` | **`intrabar_live`**(WAN-132) |
| RSI 재탭 게이트 | `first_tap_free` | **`unconditional`**(게이트 없음, WAN-123) |
| 체결 렌즈 | 3렌즈 병기 | **`baseline` 단독**(WAN-128) |
| 창 | `--years 3`(미끄러짐) | **못 박음** 2023-07-15 ~ 2026-07-15 |

🚨 **그래서 옛 표와 이 표의 차이를 어느 한 축의 몫으로 읽지 말 것** — 네 축이 **동시에**
움직였고 창까지 다르다. 이 표가 답하는 질문은 *"델타가 얼마인가"*가 아니라 **"오늘의
채택 기본값에서도 결론(순위·부호)이 같은가"** 하나다. 크기 비교는 이 표 **안에서만**
성립한다.

- 체결률이 **~75~80%** 로 나오는 것이 게이트가 실제로 꺼져 있다는 독립 증거다
  (WAN-123/124가 예고한 ~81%대 — 그전 표들은 ~50%였다).
- 심볼은 **3심볼**(BTC·ETH·SOL)이다. WAN-107/108이 판정을 낸 유니버스와 맞추기 위해서이고,
  ⚠️ [WAN-111](https://linear.app/wannabe-millionaire/issue/WAN-111)이 보인 **6심볼 희석**은
  이 표에 반영돼 있지 않다(3심볼 수치는 상단이다).

## §1 — 판정 1: 채택 TF 순위 (WAN-107) → **유지**

공식 렌즈(`baseline`) · 단일 포지션(채택 기본값) · 심볼평균:

| TF | IS | **OOS** | OOS 플러스 심볼 | OOS MDD | OOS 거래/심볼 |
| -- | -- | -- | -- | -- | -- |
| **15m** | +39.38% | **+21.10%** | **3/3** | 12.87% | 309.7 |
| **1h** | +13.56% | **+12.40%** | 2/3 | 8.05% | 84.7 |
| 4h | +10.19% | **+0.58%** | 2/3 | 3.26% | **18.3** |

- **순위 15m > 1h > 4h가 IS·OOS 양쪽에서 유지된다.** WAN-107의 결론 문장 **"15m·1h 공동
  작업 TF"** 는 오늘의 채택 기본값에서도 그대로다.
- **4h는 「보류(표본 부족)」도 유지된다** — OOS **심볼당 18.3거래**로
  [WAN-84](https://linear.app/wannabe-millionaire/issue/WAN-84) 유효 기준 20건에 여전히
  미달이다(WAN-107 당시 15.7거래). 게이트 제거로 거래가 늘었는데도 문턱을 못 넘었다.
- ⚠️ **「엣지 없음」(WAN-84/88/111/114/124)은 그대로다** — 15m OOS +21.10%는 **ETH 하나가
  끌고 간다**(ETH +43.25%, 빼면 2심볼 **+10.02%**). 1h는 더 심하다: +12.40% → ETH를 빼면
  **+0.17%** 로 사실상 0이다. 이 표는 **어느 TF 위에서 작업할지**만 확인하며 신호를 찾은
  게 아니다.

## §2 — 판정 2: 다중 대 단일의 정반대 부호 (WAN-103 §A/§C · WAN-108) → **유지**

같은 셋업 풀(체결률·eligible 동일)에서 **배치 회계만** 바꾼 두 팔이다. 다중은
`PortfolioParams(leverage=3.0)`(= WAN-108이 발표 수치를 낸 팔), OOS 심볼평균:

| TF | single | multi(3배) | 차이 | single MDD | multi MDD | single 수익/MDD | multi 수익/MDD |
| -- | -- | -- | -- | -- | -- | -- | -- |
| **15m** | +21.10% | **+35.31%** | **+14.21%p (다중 승)** | 12.87% | 18.33% | 1.64 | **1.93** |
| **1h** | **+12.40%** | +9.40% | **−3.00%p (단일 승)** | 8.05% | 13.73% | **1.54** | 0.68 |
| 4h | **+0.58%** | +0.20% | −0.38%p (단일 승) | 3.26% | 4.31% | 0.18 | 0.05 |

- **부호가 갈리는 구조가 그대로다**: 15m은 다중이, 1h·4h는 단일이 이긴다. WAN-108의
  *"두 작업 TF가 정반대를 가리키므로 하나의 기본값으로 둘 다 좋게 할 수 없다"* 는
  **기본값 유지 근거가 유지된다.**
- **위험조정에서도 방향이 같다** — 15m은 수익/MDD도 다중이 앞서고(1.93 vs 1.64), 1h는
  단일이 두 배 이상 앞선다(1.54 vs 0.68). MDD는 **세 TF 전부** 다중이 더 크다.
- **거래는 조금 늘고 수익은 크게 갈린다**(15m OOS 309.7 → 343.7건, +11%). 같은 풀에서
  회계만 달라진 결과라는 WAN-103 설계가 이 표에서도 성립한다(`eligible_setups`·체결률이
  두 팔에서 동일).
- ⚠️ **ETH 편중은 다중에서 더 심하다** — 15m OOS 다중 +35.31%는 ETH를 빼면 **+13.27%**,
  1h 다중 +9.40%는 ETH를 빼면 **−4.94%**(단일은 +0.17%). "다중이 15m에서 이긴다"는 부호는
  유지되지만 **크기는 한 심볼에 크게 기댄다.**

### ⚠️ 이 표가 확인하지 **못한** 것 — WAN-110(체결 보수화)

WAN-110은 *"15m 다중 우위가 `pen_5bp` 한 번에 85% 사라진다"* 로 기본값 유지 근거를 하나
더했다. 이 확인 격자는 **WAN-128에 따라 `baseline` 단독**이라 그 축을 다시 재지 않았다 —
즉 **§2의 15m 다중 우위(+14.21%p)는 낙관 렌즈 위의 값**이고, WAN-110의 경고는 **재확인도
반증도 되지 않았다(확인 불가)**. 필요하면 옵트인으로 언제든 잰다:
`--positions single,3 --fill baseline,pen_5bp`.

## §3 — 결론

1. **WAN-107의 「15m·1h 공동 작업 TF · 4h 보류 · 1d 제외」 → 유지.** 순위가 IS·OOS 모두
   같고, 4h의 표본 부족(18.3 < 20)도 그대로다.
2. **WAN-103/108의 「기본값(동시 1포지션) 유지」 → 유지.** 두 작업 TF의 부호가 여전히
   정반대이므로 하나의 기본값으로 둘 다 좋게 할 수 없다.
3. **뒤집힌 것이 없으므로 재-베이스라인 이슈를 제안하지 않는다.** CLAUDE.md의 "아직
   재산출 안 했다" 경고 문장은 **확인 완료**로 갱신한다.
4. **옛 리포트 md/csv는 손대지 않았다** — 당시 기록이고, `harness.LEGACY_*` 핀 덕에 여전히
   비트 단위로 재현된다. 이 문서의 표와 **열이 다르므로**(3렌즈 vs `baseline` 단독) 두 표의
   셀을 나란히 빼지 말 것.

## §4 — 도구 변경: 범용 CLI에 포지션 정책 축을 열었다

다중 포지션(WAN-103 회계)은 `PortfolioParams`를 명시로 넘기는 **파이썬 진입점에만** 있어
CLI로는 판정 2를 확인할 방법이 없었다. 새 `wanNN_*.py` 모듈을 만들지 말라는 이슈 지침에
맞춰 **CLI를 넓혔다**:

```
uv run python -m backtest.run --positions single,3
```

- `single`(기본) = 채택 기본값인 동시 1포지션 경로 **그대로**. 인자를 안 주면 예전과
  **비트 단위로 같은 행**이 나온다(회귀 테스트가 고정).
- 숫자 = 다중 포지션의 명목 상한 배수. `max_concurrent`·`one_per_zone`은 기본값 그대로라
  WAN-103/108이 발표 수치를 낸 설정과 같다.
- 행에 `position_mode`·`portfolio_leverage` 열이 생겼다. 라벨이 아니라 **동작**으로
  고정한다 — 테스트가 다중 팔에서 **실제로 포트폴리오 엔진이 호출되는지**를 검사한다
  (WAN-95/112/123 부류의 "라벨만 붙는 조용한 실패" 방지).
- 종가 진입(A안)에 `--positions`를 섞으면 **거부한다**(다중 포지션 회계는 B안 전용).
