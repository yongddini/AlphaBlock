## 동기

사용자 질문("같은 오더블록에 여러 번 진입하면 깨질 확률이 더 높아지잖아 … 재탭 없는 결과가
이미 있지 않아? 그렇게 해보자"). WAN-114 사다리에 `L0`(재탭 없음)→`L0r`(재탭 켬) 단이 있지만
**채택 기본값 조건이 아니다**(볼린저 off · RSI 게이트가 WAN-123 이전 · 오프셋 0bp). 이 이슈가
그 구멍을 닫아 **채택 기본값에서 `retap_mode`만 끄고** 손익으로 잰다.

WAN-123(RSI 게이트 제거)과 같은 부류의 「규칙 하나 빼기」 판단이다.

## 무엇을 바꿨나 — `retap_mode` 축을 범용 CLI에 노출 (코드 변경 전부)

새 리포트 모듈을 만들지 않았다(CLAUDE.md §실험 도구). `backtest.run`에 `--retap-mode` 축을
다른 축과 같은 패턴으로 추가한 것이 코드 변경의 전부다:

- `harness.RETAP_MODES = ("every_tap", "once")` · `build_params(retap_mode=...)` · `RunRow`에
  `retap_mode` 열(기본값 `every_tap` — 축이 생기기 전 행 생성부와 호환) · 표 축 열 추가.
- `run.Grid.retap_modes` 축(핀이 아니라 **진짜 축** — 두 팔을 한 표에 나란히 낸다) ·
  `Combo`·`iter_combos`·`--retap-mode` 플래그. 안 주면 채택 기본값 `every_tap` 하나만 돈다.

## 판정 — (c) 두 작업 TF가 정반대

공식 렌즈(`baseline`) 심볼평균 `total_return`:

| TF·구간 | `every_tap` | `once` | Δ | 위험조정(수익/MDD) |
| ------- | ----------- | ------ | - | ------------------ |
| 15m OOS | +8.63% | +10.16% | **+1.53%p** | 0.58 → **0.95** (once 승) |
| 1h OOS  | +5.28% | +1.25%  | **−4.03%p** | 0.53 → **0.15** (every 승) |

- **15m OOS는 `once`가 이긴다**(수익·위험조정 둘 다, MDD 14.80→10.75%).
- **1h OOS는 `once`가 진다**(−4.03%p, 위험조정 붕괴).
- IS는 두 TF 다 `once` 우위지만 1h은 **OOS에서 부호가 뒤집힌다**(IS→OOS 뒤집힘).

"TF마다 다른 재탭 정책"은 **IS에서 고르는 새 자유 파라미터**라 채택 근거로 약하다(WAN-108
논리). **기본값 `retap_mode="every_tap"` 유지.**

## 손절률은 안 움직인다 — 사용자 가설 미재현

OOS에서 팔을 바꿔도 승률(≈손절률의 여집합)·평균R은 ±1%p·±0.03R로 미동하는데, 거래 수는
~30% 줄고 `total_return`만 여러 %p 움직인다(15m OOS: Δret +1.53%p vs Δ승률 +0.57%p · Δ거래
−29%; 1h OOS: Δret −4.03%p vs Δ승률 −1.01%p · Δ거래 −31%). **재탭 거래가 유독 더 잘 깨지는
것이 아니라 같은 품질의 거래가 더 적어질 뿐이다.** 사용자 원 가설("여러 번 진입하면 깨질
확률↑")은 채택 기본값 손익 축에서 재현되지 않는다 — 이슈 보조 근거(회차별 손절률 비단조 ·
OOS 유의성 소멸)를 손익으로 재확인.

## 인용 경고

- **「엣지 없음」(WAN-84/88/111/114/124) 불변** — 네 셀 모두 ETH를 빼면 0 근처/마이너스다
  (15m OOS `every_tap` −ETH −1.81% · `once` −ETH +0.42% · 1h OOS 둘 다 마이너스). 규칙을 빼
  자유도를 줄인 측정이지 신호를 더한 게 아니다.
- **「`once`가 좋다」로 인용 금지** — 15m 한정이고 1h은 정반대다. OOS 우위가 TF에 통째로 의존.
- **비트 재현**: `every_tap` 팔이 채택 기본값 수치(15m OOS +8.63%·−ETH −1.81% · 1h OOS +5.28%)를
  그대로 낸다 — 축 추가가 기본 실행을 흔들지 않았다(회귀 테스트가 고정).

## 범위 밖 (별도 후속)

**존 단위 생애 추적**("존이 평균 몇 번 탭되고 몇 번째에 죽는가") — 거래 단위가 아니라 존
단위 라벨링이 필요하고 생존 편향을 걷어낼 수 있다. 새 파이프라인이 필요해 이 이슈(CLI 격자
한 판)와 성격이 다르다.

## 기본값·토대

바꾸지 않았다(`retap_mode="every_tap"` 유지 · `ALPHABLOCK_LIVE_TRADING=false` 유지). 측정 전용.
