# WAN-128: [재-베이스라인] 체결 민감도 렌즈 병기 폐지 — 신규 리포트는 `baseline` 단독

**결정일** 2026-07-19 · **결정 주체** 사용자 · **성격** 명시적 재-베이스라인
([WAN-104](https://linear.app/wannabe-millionaire/issue/WAN-104) 확정 렌즈 유지 · 병기 요구만 폐지 · CLAUDE.md 「토대 확정」 2번 개정)

**재현**

```
# 신규 리포트 기본 실행 = baseline 단독 (엔진·파라미터 불변)
uv run python -m backtest.run --tf 15m --symbol BTCUSDT --oos

# 폐지된 것은 "병기 요구"일 뿐, 민감도 렌즈는 옵트인으로 그대로 돈다
uv run python -m backtest.run --tf 15m --symbol BTCUSDT --oos --fill baseline,pen_5bp,pen_5bp_drop_50

# 기존 리포트 CSV 비트 단위 재현(파라미터를 안 지웠으므로 재현돼야 한다)
uv run pytest tests/test_harness.py tests/test_run_regression_real_data.py -q
```

## 요약 — 결론 먼저

1. **체결 민감도 렌즈의 「병기 요구」를 폐지한다.** 앞으로 새 리포트·결정 이슈는 공식 렌즈
   `baseline`([WAN-104](wan104.md)) **단독**으로 낸다. `pen_5bp`(관통 벌점)·`pen_5bp_drop_50`
   (스트레스)을 함께 적으라는 관례가 사라진다. 사용자 결정: *"이제 이 옵션은 다 버리면
   안돼?"* → PM이 위험을 설명하고 4개 선택지를 제시 → 사용자 *"둘 다 버리고 baseline만."*

2. **리포트 층위의 변경이다 — 엔진은 안 건드린다.** `BacktestConfig`의
   `fill_penetration_bps`(기본 `0.0`)·`fill_dropout_rate`(기본 `0.0`)·`fill_dropout_seed`와
   `backtest.run --fill` 옵션은 **전부 존치**한다. 기본값이 이미 0이라 "안 돌리면 그만"이고,
   파라미터를 지우면 과거 리포트 재현이 깨지고 되돌리기 어렵다. 두 민감도 렌즈는 **삭제가
   아니라 옵트인**이라 `--fill`로 언제든 수동 확인할 수 있다.

3. **강제 코드 경로는 애초에 없었다.** CLI 기본 실행은 이미 `baseline` 단독이다
   (`harness.BASELINE_FILL` · `run._fills_from_args`의 무인자 분기 → `(BASELINE_FILL,)` ·
   `build_params(fill=BASELINE_FILL)`). 3렌즈는 각 `wanNN_*.py` 리포트 모듈이 자기 안에서
   `LENS_NAMES = ("baseline", "pen_5bp", "pen_5bp_drop_50")`로 선언하는 **그 리포트만의
   상수**일 뿐, 공용 층에서 3렌즈를 자동으로 뱉는 경로는 없다. 따라서 "제거"할 코드는 없고,
   관례를 **문서(CLAUDE.md)와 코드 앵커 주석(`harness.BASELINE_FILL` 옆)에 못 박는 것**이
   이 이슈의 실질이다.

4. **기존 리포트 md/csv는 무수정 보존한다.** 3렌즈로 적힌 당시 기록이다. 손대면 CSV 비트
   재현이 깨진다(= 뭔가 잘못 건드린 것). 회귀 테스트가 이를 고정한다.

## ⚠️ 이 결정으로 잃는 것 (기록 — 나중에 "왜 안 보이지"를 막기 위해)

PM이 반대 근거로 제시했고 사용자가 알고 선택했다. **결정을 뒤집자는 게 아니라 잃은 것을
명시적으로 남겨 두는 것이다.**

1. **`baseline`이 낙관이라는 성질은 사라지지 않는다.** 큐 우선순위를 모델링하지 않으므로
   "닿으면 무조건 체결"로 본다. 앞으로 모든 공식 수치는 **상한**이고, 그 상한이 얼마나
   부풀려졌는지 자동으로 볼 장치가 없어진다.

2. **채택 엔진의 체결 취약성 실측치가 리포트에서 사라진다**([WAN-124](wan124.md) 3단, 실측
   `backtest/wan123_fill_conservatism.py`): 15m
   OOS에서 관통을 요구하면 **거래는 94.29%가 남는데 수익은 3.30%만 남는다**(1h는 95.46% /
   35.10%). 즉 수익 대부분이 "스치듯 닿고 되돌아선 체결"에 실려 있는데, 앞으로 이 사실이
   리포트에 나타나지 않는다.

3. **결론을 가른 선례들.** WAN-88([`reports/wan88_long_only_summary.md`](../../backtest/reports/wan88_long_only_summary.md))의 `baseline` 유의 4셀은 `pen_5bp` 한 번에
   전멸했고, [WAN-124](wan124.md)는 11→10셀로 살아남아 **"이번 건 성질이 다르다"** 는 판정이
   가능했다. [WAN-120](wan120.md)은 `pen_5bp`에서 부호가 뒤집히는 걸 보고 [WAN-119](wan119.md)
   의 잘못된 주장 하나를 걷어냈다. 이런 교차검증 축이 신규 리포트에서 기본으로는 안 돈다.

4. **체결률 81%가 전부 `baseline` 위의 값이다**([WAN-123](wan123.md) 게이트 제거 후). 낙관
   가정의 비중은 줄지 않고 **늘었다**.

5. **과거 리포트와의 대조 축이 끊긴다** — 기존 md/csv는 3렌즈로 적혀 있어 신규
   `baseline` 단독 리포트와 열이 안 맞는다.

## 「엣지 없음」은 뒤집히지 않았다

렌즈 병기를 그만둔 것이지 새 수치를 낸 게 아니다. WAN-84/88/111/114/124의 「엣지 없음」
판정은 그대로이고, 실제 체결 해소는 틱·호가 데이터([WAN-98]) 소관이다.

## 선택 사항 — 「체결 취약성 한 줄」은 넣지 않았다

이슈는 선택 사항으로 *"리포트마다 채택 기본값·공식 TF·OOS 1셀에 대해 `pen_5bp` 잔존
수익률 % 한 줄"* 을 제안하면서, **사용자 결정 없이 개발자가 임의로 넣지 말라**고 명시했다
(*"결정 없이 개발자가 임의로 넣지 말 것."*). 이 작업은 매시간 자율 루틴에서 수행됐고 그
결정 입력을 받을 수 없어, 보수적으로 **미구현**했다(추가 = 명시적 사용자 결정 필요). 원하면
언제든 별도 이슈로 붙일 수 있고, 근거 데이터(`pen_5bp` 잔존율)는 `--fill`로 이미 산출 가능하다.
